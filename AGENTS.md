# 项目Agent开发基线

## 1.项目信息、技术栈版本

- 项目路径：`/Users/se7en/ArtProject/platform`
- 项目定位：美术部门生产工作台，用于禅道任务/Bug 同步、任务中心、AI产物清单、成员产物、质量评分、库存缓存、美术执行台、本机直接执行状态、AI档案和负责人管理操作。
- 默认回复语言：中文。
- 默认用户身份：美术部门负责人。涉及任务、排期、周报、资源协调、风险识别、权限配置、组员管理、Worker 状态、AI产物清单和执行状态时，优先按工作台美术负责人管理者视角组织回复。
- 术语边界：
  - `admin`、`美术负责人`、`负责人管理者`、`工作台管理者` 指工作台内具备管理权限的美术负责人账号。
  - `执行人`、`组员`、`被指派人` 指实际领取并执行任务的人。
  - `任务负责人`、禅道原始负责人不得与工作台 `admin` 混用。
  - 用户说“我”“负责人”“美术负责人”“负责人管理者”“工作台管理者”时，默认指工作台里的美术负责人管理者 `admin`；在本地执行状态、Worker 状态、执行状态和权限语境里同样适用，不是某条任务的执行人，也不是禅道任务原始负责人。
  - 用户说“基线”时，默认按本文件要求更新 `AGENTS.md`：只保留已确认硬性约束、功能边界、数据结构、接口约定、权限约定、禁用写法和固定命令；不得写入调试代码、临时方案、作废方案或闲聊内容。
  - 用户要求“汇总本次全部会话内容”“输出标准 AGENTS.md 全文”“后续新会话自动加载遵守”时，必须直接更新本文件；若本文件已存在，必须结合旧规则和本次新增规则提炼，不得简单覆盖或删除已确认硬约束。
  - 当前会话内容较多、接近上下文压缩、或已经形成稳定的新界面/新操作/新权限/新数据约定时，即使用户没有再次完整复述，也应主动按“基线”口径更新本文件。
- 用户可见文案、说明文档、标记文档、看板文案统一使用中文。代码标识、命令、路径、文件名、字段名、接口参数、第三方专有名词可保留英文。
- 全项目默认改动边界：
  - 没有被负责人明确要求处理的模块、字段、样式、交互、脚本、接口、数据和流程，一律不得擅自变更。
  - 已经被负责人强调保持原样的内容，除非负责人再次明确说要改，否则必须继续保持，不得借其它功能顺手调整。
  - 每次对话里的改动都必须只围绕负责人当次明确要求的点执行到位；不得因为实现方便顺手修改其它已完成、已确认或未被点名的功能。
  - 如果实现某个需求会牵连其它模块、字段、数据链路、展示逻辑、缓存、统计、权限、接口或交互，必须先向负责人说明牵连点、影响范围和必要性，得到明确确认后再执行。
  - 当确认某项改动不会影响其它已做好的功能时，应直接按要求完成，并保持原有扫描结果、缓存数据、展示口径、关联数据和其它界面行为不变。
  - 任何“改 A 导致 B 出问题”的情况都必须视为违规；例如修改 AI 产物刷新逻辑时，不得影响已扫描展示内容、版本、贡献人、调用次数、有效占比、质量分、别名、分类、统计卡、筛选列表、成员端展示或其它页面。
  - 所有功能改动必须以性能优先，默认使用轻量读取、分页/虚拟化、缓存快照、增量更新和按需加载，保证页面切换不卡顿、内容不消失、展示不闪烁。
  - 一级菜单、链接和路由切换必须先完成视图切换，再异步补齐当前页面所需的轻量数据；不得在点击链路里同步恢复大缓存、重扫库存、重算评分、渲染整段日志或重复触发同一页面的数据加载。
  - 重复点击当前页面入口必须短路或只做必要状态校正，不得重复写浏览器历史、重复恢复缓存、重复发起接口刷新或导致页面卡顿。
  - 页面自动补数据和平台事件刷新必须区分“当前页面”和“非当前页面”：当前页面只做轻量刷新，非当前页面只标记待更新，等负责人真正进入该页面后再补齐，不得在后台同时刷新多个重页面。
  - 自动后台刷新必须有并发去重和合理冷却；同一类数据已有请求在途时复用在途请求，短时间重复事件不得连续打接口。负责人手动点击同步、刷新、保存或删除后的真实刷新不得被后台冷却拦截。
  - 任务、执行记录、验收记录、Worker 状态、AI档案等高频渲染列表的关联查询必须优先使用 `Map`、分组索引、分页或虚拟化；不得在每一行渲染时反复对全量数组 `filter/find/sort` 造成切页卡顿。
  - 数据类界面默认展示已有服务端缓存或已落盘快照；除非负责人点击该界面的刷新按钮、相关源文件确实新增/修改/删除并影响依赖，或该功能有明确联动依赖，否则不得重新计算分数、重扫文件、重建列表或覆盖缓存。
  - 任务中心、AI产物清单等数据页刷新网页时，首帧必须优先展示上次可信缓存或首屏展示快照；不得先显示 0、空列表、空白或只有局部统计，再跳成上次缓存数据。
  - AI产物清单首屏快照只允许作为展示缓存，用于普通刷新和首次进入时先显示上次第一页清单；不得把快照写回扫描源、不得覆盖调用次数、有效占比、质量分、版本、贡献人、别名或分类等真实数据。
  - 平台必须保持轻量级数据模型；不得因为日志、临时事件、轮询、上报、刷新或补计数无缘由累加长期数据。
  - 所有界面只要出现 `删除` 按钮或删除接口，必须真实硬删除该入口所属数据源中的对应记录、文件、缓存或工作区，不得只做前端隐藏、软删除、标记删除或继续占用后台/数据库/本机内存；功能语义本身是 `作废`、`隐藏`、`禁用`、`恢复` 的入口除外。
  - 每个界面的删除入口只能删除本界面或本业务域明确拥有的数据；如果某类数据已经有独立删除入口，大范围删除、上级来源删除、日志删除、扫描源删除、项目删除、缓存清理或自动 retention 不得越权强制删除它。
  - 删除有关联数据时，只能断开或清理当前业务域内的派生数据，不得影响其它业务事实数据的计算口径；例如删除任务中心平台单不得删除已产生的执行记录，删除操作日志不得回退 AI产物使用次数，删除扫描源不得删除美术执行台记录。
  - 增长型累计指标必须按“只增不因明细删除减少”处理；删除操作日志、AI档案明细、扫描来源、项目接入记录、原始上报明细或短期缓存，都不得扣减、清零、重算或删除 `usage-counters.json` 中已累计的调用次数、有效人数和历史桶。
  - 如果确实需要彻底清理某个已有独立删除入口的数据，必须走该入口自己的按钮和 API；不得借其它范围删除、后台清理或上级删除顺带完成。
- 清理维护规则：
  - 项目清理前必须先执行只读巡检 `npm run audit:cleanup`，按巡检输出区分“可直接清理”“建议负责人确认后再清理”和“禁止手动删除”。
  - `维护中心` 是管理员专用系统维护页面；前端菜单必须同时限制 `admin` 和 `menu.maintenance`，后端 `/api/maintenance/*` 必须同时校验 `admin` 和 `api.maintenance.manage`。
  - 维护中心只负责可视化展示本机数据体量、记录数量、删除前预览和确认后硬删除；不得在进入页面时自动清理、自动压缩、自动重算或后台删除任何业务数据。
  - 维护中心所有清理动作必须先调用预览接口，展示命中数量、预计释放空间和样例对象；负责人修改清理类型或筛选条件后，原预览必须失效，必须重新预览后才允许确认删除。
  - 维护中心确认删除只能按当前预览一致的业务域执行硬删除，不得因为上级范围、日志范围、安全清理或摘要清理顺带删除其它页面已有独立删除入口的数据。
  - 维护中心 `安全维护清理` 只允许处理 `.DS_Store`、过期 JSON 临时文件和 `runs.json` 已不存在的孤儿执行工作区；不得删除业务 JSON、有效执行工作区、正式产物、扫描源、任务、禅道数据或累计指标。
  - 维护中心 `操作日志范围删除` 必须选择开始时间和结束时间；只删除操作日志数据源中的命中记录，不得回退、扣减、清零或重算 `usage-counters.json`。
  - 维护中心 `AI档案 / 执行记录删除` 必须选择开始时间和结束时间；只能删除非运行中的执行记录、对应执行工作区和产物归档目录，不得删除任务中心任务、禅道任务、扫描源、AI产物源文件或累计调用指标。
  - 维护中心 `美术摘要产物清理` 必须选择时间范围、任务号或关键词之一；只删除 `outputs/art-briefs/` 下命中的摘要目录，并同步 `data/art-briefs.json` 和摘要索引，不得清理其它 `outputs/` 内容。
  - `.DS_Store` 属于无业务意义的系统元数据，可直接清理；如果 `npm run smoke` 或外部 Git 缓存扫描后重新出现，应再次按巡检清理，不得因此修改扫描业务逻辑。
  - `logs/` 只能在排障窗口结束后按日期归档或截断；截断时必须保留原日志文件路径，避免 LaunchAgent 后续写入失败。
  - `data/restore-backups/` 是历史修复前备份目录；确认对应修复稳定后只能归档到 `data/restore-backups-archive/` 或外部备份，再清空散文件，不得删除 `data/` 其它业务 JSON。
  - `data/`、`workspace/`、`workspace/artifacts/`、`outputs/art-briefs/`、`dist/`、`node_modules/` 属于受保护目录；不得为“清理空间”手动删除，必须走对应业务入口或安装/构建流程。
  - 外部地址、本机路径和环境变量必须先登记在 `docs/外部配置登记.md`，再逐条配置化；不得一次性替换禅道、Google Sheet、组员上报、Git 仓库或本机工具路径默认值。

## 2.命名、组件、样式、目录规范

- 保持现有目录、模块边界和已确认交互流程；不得为了局部修复顺手重构、改名、换布局或调整业务结构。
- 关键文件职责：
  - `server/server.mjs`：平台接口、缓存读取、禅道同步入口、权限判断、本机 Worker 状态和直接执行任务接口。
  - `server/store.mjs`：本地 JSON 数据读写、执行记录、Worker 心跳、直接执行领取和状态回传。
  - `server/auth.mjs`：账号、角色、权限目录、默认角色和历史权限兼容。
  - `server/scanner.mjs`：Git、本地路径、共享盘扫描与产物内容解析。
  - `src/App.vue`：全局业务状态、路由守卫、权限判断、库存缓存读取、刷新库存、贡献人覆盖、AI评分、产物使用统计、直接执行创建和 Worker 状态读取。
  - `src/views/SkillInventoryView.vue`：AI产物清单、贡献人调整、使用明细抽屉、成员产物整合入口。
  - `src/components/dialogs/SkillPreviewDialog.vue`：Skill/md 内容预览、调用别名维护和直接执行到 Figma 的创建入口。
  - `src/components/dialogs/RunCreateDialog.vue`：新建美术执行弹窗。
  - `src/views/RunsView.vue`：美术执行台、执行详情、直接执行的本机状态提示。
  - `src/views/AgentWorkersView.vue`：本机执行状态页，展示 Worker 心跳、自检和执行明细。
  - `src/views/AiArchiveView.vue`：AI档案页，展示完整 AI 执行明细、筛选和范围删除。
  - `src/views/AiMembersView.vue`：AI部门看板页，展示负责人/组员 AI 看板、当月 AI 评分和缓存后的部门 AI 进展 HTML 看板。
  - `src/views/RoleManagementView.vue`：角色管理和细粒度权限配置。
  - `src/styles.scss`：全局展示兜底样式、抽屉、评分面板、弹窗分段控件样式。
  - `scripts/art-direct-worker.mjs`：组员本机直接执行 Worker。
  - `scripts/install_art_direct_worker_launch_agent.sh`：安装组员本机 Worker 开机自启。
  - `scripts/git-auto-backup.sh`：Git 自动备份执行脚本。
  - `scripts/install_git_auto_backup_launch_agent.sh`：Git 自动备份 LaunchAgent 安装脚本。
- 工作台不是营销页或展示页。UI 优先保证数据准确、信息完整、操作稳定、密度合理。
- 侧栏品牌区规则：
  - 工作台侧栏必须默认完整展示，不得提供收起/折叠按钮，不得通过本地缓存恢复为窄侧栏。
  - 响应式宽度变化时，侧栏菜单文字和品牌文案不得自动隐藏成只剩图标；确需移动端适配时必须另行设计，不得复用桌面收起态。
  - 侧栏主标题固定为 `美术部工作台`；副标题 slogan 固定为 `美术生产与AI协作中枢`，必须单行完整展示。
  - slogan 中 `AI` 左右不得加空格，不得写成 `美术生产与 AI 协作中枢`。
- 工作台关键业务文本不得无意义省略，尤其是按钮、分段控件、标签、抽屉标题、成员名、产物名、版本记录、任务标题和操作字段。
- 长路径、日志、表格大段文本可以受控换行或提供详情入口，但不能让负责人无法判断真实内容。
- 列表操作栏规则：
  - 任务中心操作栏固定在右侧。
  - AI产物清单操作栏固定在右侧。
  - 响应式宽度变化时操作字段不得消失。
  - 操作按钮必须左右横向排列，不得换行导致按钮不可见。
  - 任务中心 `拆单` 按钮必须放在右侧操作区，和 `发起执行` 并排展示；不得放在任务标题旁边挤占标题内容。
- 全局右侧半屏抽屉：
  - 右侧半屏抽屉必须为直角表现。
  - 所有通过 Element Plus `ElDrawer` 从右侧滑出的界面都必须全局直角，包含抽屉本体、头部和内容区。
  - 不得用 `.el-drawer.rtl`、`.art-progress-log-drawer`、`.skill-usage-drawer` 或类似选择器恢复圆角。
  - 通过 `append-to-body` 渲染时，样式必须同时考虑全局样式和组件内 scoped 样式。
- “新建美术执行”弹窗：
  - 执行来源和执行范围分段控件必须使用全倒角选中底图。
  - 文字必须完整可见。
  - 不得改变已确认交互流程。
  - 执行来源只保留 `独立执行` 和 `关联任务`，顺序必须为 `独立执行` 在左侧并默认选中，`关联任务` 在右侧；不得再要求选择项目。
  - `负责人 / 执行人` 必须显示在执行来源右侧，显示当前点击创建执行的账号展示名，并且不可输入修改。
  - 美术执行方式必须直接展示可选项，不得使用下拉框；选项包含 `单skill/md`、模板库里已保存的自定义流程模板名称，以及保留原本的 `自定义流程` 临时多步骤入口。
  - `Figma 链接` 是新建美术执行的必填项，代表本次执行要处理的具体 Figma frame、分区或整页链接。
  - 选择 `单skill/md` 后，下方必须沿用原本的单个 md / Skill 选择能力，再展示 `Figma 链接` 和 `给 Codex 的执行要求`。
  - 选择某个已保存模板时，不再要求选择具体 md / Skill，必须直接使用模板里已保存的步骤；下方只保留 `Figma 链接` 和 `给 Codex 的执行要求`，不得再额外展示 md/Skill 选择下拉或 `使用已保存模板` 下拉。
  - 选择某个已保存模板时，不得把模板说明自动填入 `给 Codex 的执行要求`；模板说明只用于模板管理和选项摘要。`给 Codex 的执行要求` 默认保持空，由创建执行时按 Figma 链接和当前执行方式生成默认要求。

  - `继续执行` 和 `重新执行` 必须保持语义分离：继续执行复用当前记录，重新执行新建记录；不得把失败/阻塞后的主按钮做成重新执行，也不得把重新执行做成清空当前记录。
  - `继续执行` 不等于恢复已被 kill 的 Codex 进程内存；它必须通过读取已有产物、阶段报告、旧日志尾部和当前 Figma 状态来续跑，避免重复创建已完成节点或重复执行已完成步骤。
  - 每条美术执行记录必须长期保留 `startedAt`、`finishedAt`、`durationMs`、`stages[].startedAt`、`stages[].finishedAt`、`stages[].durationMs` 和必要阶段状态；除非负责人通过对应业务删除入口明确删除该执行记录，否则刷新、切页、操作日志删除、AI档案筛选或轻量恢复都不得清空这些字段。
  - 本机执行状态不得只看 `claimedAt`、`startedAt` 或 `status=running` 判定为真实执行。必须结合 Worker 心跳 `currentRunId`、服务端 `run.log`、`workerLocalLogPath/workerLocalLogSize`、`workerResult`、`resultSummary`、`figmaWriteResult`、产物目录和最近真实活动时间共同判断。
  - Worker 已领取并回传 running，但长时间没有 Codex stdout/stderr、没有最终状态、没有 Figma 写入证据，且领取设备在线但 `currentRunId` 已为空或不等于该执行时，平台必须判为 `本机回传失联` / `本机阻塞`，不得继续显示普通 `本机执行中`。
  - Worker 已领取后领取设备心跳超时、失联或没有可用心跳记录，同时没有 Codex 输出、最终状态或产物证据时，也必须判为 `本机回传失联` / `本机阻塞`；不得因为 Worker 不在线就跳过对账，长期保留 `running` / `claimed`。
  - 日志抽屉没有 `run.log` 时必须返回可读诊断，说明领取设备、领取时间、开始时间、最近状态回传、本机日志路径和阻塞原因；不得只显示“暂无关键执行日志”让负责人误判为仍在正常实时接收。
  - Worker 自更新后必须先退出或重启，不得在已安排自更新重启的同一轮继续领取或恢复新任务，避免刚领取就退出造成空日志 running。
  - Worker 恢复旧 `running` 记录必须有本机未完成快照和可恢复证据；如果本机日志只有 Worker 启动提示、没有任何非 `[worker]` 的 Codex 输出，且已超过无输出阈值，必须清理本机旧状态并交给平台判阻塞或由负责人明确继续执行，不得反复恢复同一条旧任务。
  - Worker 启动 Codex 后必须尽早回传本机日志路径和日志大小；如果 Codex 长时间没有任何 stdout/stderr 输出，必须超时终止并回传失败/阻塞摘要，不得无限保持 running。
  - 执行记录主状态已经进入 `completed`、`blocked`、`failed`、`cancelled` 等终态后，`workerStatus` 不得继续保留 `claimed`、`running`、`queued`、`pending` 等活动态；服务端读取时必须自动归一，前端展示也必须优先尊重主终态，避免负责人看到“已中断但本机执行中”的矛盾状态。
  - 美术执行台右侧 `执行步骤明细` 必须展示累计执行耗时和每步阶段耗时；累计耗时显示在模块右上角，每步下方按上下布局展示状态和耗时，例如 `已完成`、`00:00:00`。已有真实耗时不得因刷新、读取阶段报告、恢复历史记录或界面重算变成 `00:00:00`。
  - 中断、继续执行和重新执行都不得无故丢失已有阶段耗时。`继续执行` 只能从未完成阶段追加或更新耗时；`重新执行` 必须在新执行记录里重新计算阶段状态和耗时，原执行记录已有耗时、日志和结果必须保留。
  - 美术执行台右侧任务明细必须使用统一基础结构：顶部标题和操作按钮、`执行步骤明细`、结果明细模块、执行对象、执行环境、问题提示、下一步操作和 `查看档案` 入口。不同执行类型只能在这套基础结构后面新增补充模块，不得替换成另一套主展示。
  - 普通“新建美术执行”或分阶段执行可以额外展示关键动作概览、任务链路和继续和 Codex 沟通，但这些只能作为结果明细后的补充内容，不能替代统一的任务明细基础结构。
  - 只要执行记录已经明确选择了某个 md、SKILL.md 或 Skill 作为本次执行依据，右侧不得再展示关键动作概览、任务链路、md/SKILL.md 引用汇总或“明细在产物列表查看”这类重复说明；这些内容属于创建时已选定信息，不是执行中的有效明细。判断时必须同时看创建字段和执行链路解析结果，例如 `primarySkillPath`、`selectedMaterialHints`、`showdocHints`、`stage`、`executionMode === single-skill`、`workflow === art-single-skill`、`direct-skill`、以及 `runReferenceCount(run) > 0`，不得只看单一字段。
  - 引用单个 md/Skill 的直接执行（`direct-skill`）右侧只展示 Worker 状态、轻量阶段进度、直接执行结果、执行对象、执行环境、问题提示和 AI档案入口；不得展示关键动作概览、任务链路、查看具体明细式链路文案或继续和 Codex 沟通入口。
  - 在“新建美术执行”里已选择 md/Skill 的单技能执行（`art-single-skill`）也不得展示关键动作概览；右侧必须按正常操作流程展示执行步骤、结果明细和日志。
  - 单 md/Skill 执行右侧标题说明必须使用 `查看当前任务的执行步骤明细` 口径。
  - 如果任务链路本质是在表达步骤流程，单 md/Skill 执行时必须上移到当前状态下方、阶段进度附近，标题使用 `执行步骤明细`；不得放在关键动作概览下方作为独立下沉模块。
  - 单 md/Skill 执行步骤必须使用横向流程步骤图展示，按 `选择执行内容`、`读取任务上下文`、`执行自检`、`Codex 执行`、`结果回传`、`归档验收` 这类负责人可理解的步骤呈现；当前运行步骤必须高亮，已完成、失败、待执行要有清晰状态。
  - 单 md/Skill 执行结果必须模块化展示交付判定、成功数量、失败数量、扫描点、数据类、执行对象、执行环境、问题提示和 `查看档案` 入口；不得默认重复展示旧的纵向任务链路或关键动作概览。
  - Web 端 Codex 对话入口必须做成 Codex 官方图标悬浮按钮，不得作为右侧详情里的固定卡片或执行过程中的常驻面板。
  - Web 端 Codex 悬浮入口只在用户点击左侧美术执行清单中的某条非 `direct-skill` 执行记录后出现；出现后必须作为当前执行台窗口内的页面级固定悬浮入口常驻显示，不得挂在左侧任务条上、不得只在任务条附近或 hover/焦点状态下临时显示，也不得被限制死在右侧明细卡片内部。
  - Web 端 Codex 悬浮入口关闭时只收起对话面板，不得连同悬浮图标一起隐藏；未选中左侧记录、当前执行正在运行、或记录是 `direct-skill` 时必须隐藏入口。
  - Web 端 Codex 悬浮入口允许在当前页面拖动位置；拖动位置只作为浏览器本地 UI 状态保存，不写入后端，不产生额外数据记录，拖动后必须默认回到上次位置。
  - Web 端 Codex 悬浮入口拖动、hover、进入右侧执行过程区域、切换滚动位置或展开/收起对话面板时，不得改变悬浮图标位置或导致入口消失；位置变化只能来自用户拖动或视口尺寸变化后的边界收敛。
  - 点击 Web 端 Codex 悬浮入口后，打开的对话只对应当前左侧选中的执行记录；提交追加要求必须创建新的执行记录继续处理，不能把要求写到其它任务上。
  - Web 端 Codex 对话窗口必须默认收起；单 md/Skill 执行不允许直接展开对话窗口。展开后才渲染输入框、模型、推理强度和请求标准。
  - Web 端 Codex 对话提交后创建新的执行记录，不在当前记录里追加长对话历史；默认不得读取或渲染历史长日志、整段模型事件或大段上下文。
  - Web 端 Codex 对话可选择模型和推理强度；单次配置只作为新执行的 `codexRequest` 保存和启动参数使用，不得保存 API Key 或 Figma OAuth token。
  - 直接执行如果后端没有阶段数据，右侧阶段进度必须兜底展示 `已创建`、`Worker 领取`、`本机执行`、`结果回传`，让负责人能按步骤判断状态。
  - 分阶段普通任务仍必须按阶段一步步展示，适合使用横向阶段步骤呈现；不得因直接执行轻量视图而删除普通分阶段任务的阶段进度。
- `/agent-workers` 本机执行状态页：
  - 必须作为一级页面展示，不得藏在深层入口。
  - 页面必须区分工作台管理者和组员美术执行者。
  - 管理者可见完整的直接执行流程说明、管理判断文案和给组员分发命令的说明。
  - 非管理者不得看到页面开头“本机执行状态 / 查看组员 Worker 在线...”整条提示、`直接执行启动方式` 模块、管理者口径的 `本机 Worker 绑定与实验操作` 文案。
  - 组员必须能复制自己的 `复制手动启动` 和 `复制开机自启` 命令。
  - `Worker 心跳` 是固定模块名，不得改回 `组员本机 Worker`。
  - `Worker 心跳` 模块标题下只保留“命令必须在执行人自己的电脑运行。”这类操作提示。
  - `Worker 心跳` 模块头部右侧必须放 `复制手动启动`、`复制开机自启`、`刷新状态`，按钮左右横向排列。
  - `刷新状态` 按钮只代表负责人或组员手动刷新本机执行状态页；后台实时事件、心跳变更、执行记录变更和静默补数据不得共用该按钮 loading，不得造成按钮一直转圈或让负责人误判为 Worker 高频加载。
  - `Worker 心跳` 模块不得放顶部六个统计小块；在线、就绪、待领取、执行中、已完成等信息必须并入每个组员卡片。
  - `Worker 心跳` 必须按可见组员展示 Worker 信息；没有心跳的可执行组员也要展示；组员账号也能看到全部可见组员卡片。
  - 卡片里的组员名称必须优先使用平台账号展示名绑定 Worker `userId`；不得只显示 `admin`。
  - 卡片内必须展示组员名称、设备信息、Codex 是否就绪、Figma MCP 是否就绪、最近心跳、待领取数量、执行中数量、已完成数量。
  - Worker 设备允许本人维护展示花名；花名只影响工作台展示，不覆盖真实 `deviceId`、真实设备名或心跳绑定关系。
  - Worker 设备展示必须优先使用本人维护的花名；`Worker 心跳` 卡片、美术执行台直接执行详情、执行明细里的领取设备都必须同步展示花名，且所有成员可见同一展示。
  - `Worker 心跳` 的心电图表现以“该组员本机是否可以直接执行”为依据：只有 Worker 在线且 Codex 与 Figma MCP 都可用时，才显示绿色动态心跳；没有心跳或任一能力未就绪时，显示红色直线。
  - `执行明细` 是固定模块名，不得改回 `直接执行队列`。
  - `/agent-workers` 的 `执行明细` 只展示最近 10 条执行工作台记录，数据源必须和美术执行台的 `runs` 队列一致，不得只筛 `direct-skill`；Worker 心跳卡片里的待领取、执行中、已完成统计仍按直接执行 Worker 队列计算。完整执行明细放在左侧一级菜单 `AI档案`。
- `AI档案` 页面：
  - `AI档案` 是一级菜单，用于展示完整 AI 执行明细。
  - 必须支持关键词、人员、类型、状态、开始时间和结束时间筛选。
  - 筛选项变更后默认立即生效，不需要额外 `查询` 按钮。
  - 关键词搜索框固定放在页面标题卡片右上角。
  - 表格执行内容只展示具体使用的产物/Skill/md 名称和类型，不展示本地路径。
  - 执行内容不使用按钮式底色和描边，只保留可点击文本表现。
  - 页面顶部必须展示当前筛选范围内的 `归档任务`、`已闭环`、`待返工`、`待验收` 四项汇总。
  - 顶部四项汇总必须表现为明显分开的四块区域，并且支持点击筛选；点击后下面执行列表必须精准展示对应类别记录，`归档任务` 表示当前筛选范围内全部记录。
  - 每条执行记录必须能在 AI档案当前页面直接查看明细，不得因点击明细跳回美术执行台或任务中心。
  - 明细必须展示该次执行的成功数量、失败数量、扫描点数量和具体数据类；数据优先来自 `run.resultSummary`、`stages`、`changeSummary`、验证命令、产物证据和 Figma 链接等结构化字段。
  - 执行明细必须按负责人易读方式组织，优先展示交付结论、执行对象、执行环境、问题与待处理、本次检查内容、产物与证据；不得重复展示同一字段，也不得默认展示 `thread.started`、`item.completed` 等模型事件串。
  - AI档案明细默认不得读取或渲染整段原始日志；原始日志只能按需、轻量、尾部读取，避免点击查看档案卡顿。
  - AI档案列表必须分页或虚拟化，默认只渲染当前页，避免执行记录变多后切换页面和打开明细卡顿。
  - 支持按当前筛选时间范围删除明细；删除必须清理后端 `runs` 执行记录、对应工作区和产物目录，不得只是前端隐藏。
  - 删除明细不得回退或清零 AI产物清单累计调用次数。
- 操作日志页面：
  - 操作日志只记录负责人需要追溯的关键操作，例如创建/启动/重新执行/中断/删除执行记录、AI档案范围删除、删除业务数据、权限/角色/配置变更、任务拆分/指派/量级调整、AI产物作废/恢复/版本/别名变更等；普通登录、退出登录、登录态过期、切换页面、进入页面和页面浏览不得写入操作日志。
  - `删除当前范围` 和 `删除全部日志` 必须从后端操作日志数据源中真实移除匹配记录，不得只做前端隐藏、软删除、标记删除或写入“已删除”占位记录。
  - 操作日志删除后不得再反写一条新的“删除操作日志”审计日志；负责人删除全部日志后，操作日志列表应允许真实变为空。
  - 删除操作日志只删除操作日志明细本身，不得回退、扣减、清零或重算 `usage-counters.json` 中已经累计的 AI 产物调用次数。
  - 删除操作日志不得联动删除、重置或回滚任何已有独立删除入口的业务数据，包括 `runs` 执行记录、AI档案、任务中心、Bug 队列、人工验收、AI产物清单、Worker 心跳和使用次数；业务数据只能由对应业务入口删除。
  - 操作日志页面的删除按钮只认操作日志自己的数据源；不得误删、清空、重算或影响美术执行台、AI档案、任务中心、Worker、AI产物清单等其它页面数据。
  - 操作日志列表、详情弹窗、说明文案、搜索结果和浏览器缓存恢复必须使用同一条展示归一逻辑；同一条记录在外层列表和内层详情中对象名、描述中的对象名必须一致。凡是后端返回字段需要按业务规则修正展示名，前端渲染和缓存恢复也必须走同一修正函数，不得只修详情或只修列表。
- 美术执行台和 AI档案数据刷新规则：
  - `runs` 执行记录是动态队列数据，不等同于库存扫描缓存。
  - 进入美术执行台、`/agent-workers` 或 `AI档案` 时，必须轻量刷新 `/api/runs`，不能因为本地 `runs` 缓存已有数据就跳过后端刷新。
  - 进入 `/runs` 或刷新美术执行台时，必须先恢复或轻量加载项目目录/扫描源缓存；只有服务端项目目录已经确认加载完成后，才允许把执行记录判定为 `来源已删除`。不得因为前端本地 `projects` 尚未恢复、页面刚刷新或还没点进产物界面，就误显示 `来源已删除`。
  - 轻量刷新执行记录不得触发库存扫描、Git 拉取、本地路径扫描或共享盘扫描。
  - 进入任务中心或恢复任务中心视图时，必须轻量刷新 `/api/tasks`、`/api/bugs` 和任务验收记录；接口失败或临时返回空时保留当前展示和浏览器缓存，不得用空结果覆盖上次同步任务数字。
  - 刷新后如果当前选中的执行记录已经被后端删除，必须自动切到最新可见执行记录或清空选择，不能继续显示本地旧记录。
  - 在 `/runs`、`/agent-workers`、`/ai-archive` 这类执行相关页面登录或刷新时，只恢复当前页面必需的轻量数据，例如 `runs`、`agentWorkers` 和账号列表；不得同时恢复库存、验证、成员快照、使用统计等重数据。
  - 美术执行台里的 `查看档案` 固定进入一级菜单 `AI档案`，并按当前执行记录 ID 精确筛选明细；不得跳回任务中心。
  - 美术执行台里的 `去人工验收` 才进入任务中心并打开关联业务任务验收。
  - 美术执行台和 AI档案之间切换时，不得触发库存扫描、Git 拉取、共享盘读取、整段日志渲染或任务链路大规模重算。
  - `listRuns`、`/api/runs` 读取层可以读取产物目录中的 `阶段执行报告.md` 或同等阶段报告来补齐展示状态，但报告只允许补缺，不得用报告模板里的 `未执行`、`待执行`、空耗时或默认步骤覆盖 `runs.json` 已经存在的真实状态、起止时间、阶段耗时和累计耗时。
  - 如果某个阶段已经存在 `durationMs`、`startedAt` 或 `finishedAt`，读取层必须保留该阶段已有状态和耗时；当历史数据出现“有耗时但状态仍是 pending/未执行”时，应按完成、中断、跳过等业务语义做保守兜底，避免界面显示“未执行但有耗时”的矛盾状态。
- 本地弹窗、提醒、LaunchAgent、轮询脚本：
  - 不得造成 Dock 反复闪烁、抢焦点、重复弹窗或窗口反复出现/消失。
  - 不得使用无限重弹的 `while true` 弹窗逻辑。
  - 同一轮多条待处理事项必须串行弹窗或写入待弹队列。
  - JXA/Cocoa 自定义窗口默认使用 `NSApplicationActivationPolicyAccessory`。
  - 弹窗必须固定尺寸和稳定布局，长内容使用滚动区域、列表或摘要窗口。

## 3.接口、数据结构、数据库约定

- 禅道同步范围：
  - 只看部门 ID `27`。
  - 精准抓取美术部门人员身上的任务。
  - 任务和 Bug 必须作为独立队列分开同步。
  - 任务同步失败不得影响 Bug 队列，Bug 同步失败不得影响任务队列。
  - 当前接入的 `zentao-mcp` 任务封装只确认支持按执行读取任务列表，例如 `GET /api.php/v1/executions/:id/tasks`；除非后续验证禅道接口支持，不得假设可以按部门或人员直接查询全部任务。
  - 当前禅道 REST 实测中，`/api.php/v1/tasks?assignedTo=...` 对部门 27 账号返回空结果，`/api.php/v1/bugs` 的 `assignedTo/openedBy/resolvedBy` 查询参数会被忽略；在未验证新接口可用前，不得把这些参数当作可靠的部门人员直查能力。
- 禅道字段保护：
  - 未经负责人明确要求，不得修改或清空父任务、关联需求、任务描述、标题、截止日、模块、任务类型、状态、预计工时、剩余工时等原字段。
  - 如确实需要改这些字段，必须由负责人明确要求并二次确认。
  - 工作台拖拽指派、拆单、同步、校验、恢复、重试等内部操作不得写入禅道备注、历史备注或评论；操作留痕只允许写入平台操作日志。
  - 未提供明确输入框让负责人填写的内容，不得擅自拼接成禅道 `comment`、备注或描述提交；只有负责人明确填写“写入禅道备注/评论”的内容并确认时，才允许提交到禅道备注字段。
- 禅道同步按钮：
  - 任务大厅按钮文案为 `同步任务`，只触发任务同步和任务列表刷新。
  - Bug 大厅按钮文案为 `同步 Bug`，只触发 Bug 同步和 Bug 列表刷新。
  - 普通点击 `同步任务` 必须走快速任务同步，不得默认放大为全量历史扫描或同时同步 Bug。
  - 负责人点击 `同步任务` 或 `同步 Bug` 后，平台侧落盘的数据必须对所有已打开工作台的负责人和组员账号保持一致；同步完成后必须通过平台事件或同等轻量机制让组员端自动刷新任务/Bug 数据，不要求组员手动刷新浏览器。
  - 任务中心数字、成员卡片任务数和当前任务列表必须以服务端已落盘的 `data/tasks.json` / `/api/tasks` 同步数据为准；构建、重启、修改其它功能、页面切换、浏览器缓存恢复或平台事件刷新都不得让上次同步后的任务数字丢失或变成 0。
  - `data/tasks.json`、`data/bugs.json`、任务验收、任务备注和美术摘要属于任务中心业务事实数据，不得纳入 2 天短期 retention 清理；构建、重启、启动保留清理或后台日志清理都不得删除这些任务中心数据。
  - `data/runs.json` 是美术执行台和 AI档案的执行事实数据，不得纳入 2 天短期 retention 清理；除非负责人通过执行台删除记录或 AI档案范围删除入口明确删除，否则执行记录、状态、阶段、Figma 链接、补充要求、产物目录和日志路径必须长期保留。
  - 平台创建的任务中心单据允许负责人删除；删除必须走后端硬删除，真实移除 `tasks` 中的平台任务记录，并清理该单的任务备注、人工验收和美术摘要等任务中心附属数据，不得只做前端隐藏。
  - 通过平台任务中心创建的本地单据必须默认写入 `source: "platform"`，并补齐 `developer`、`assignedTo`、`zentao.assignedTo`、`zentao.assignedToName` 等负责人展示字段；不得写成 `manual` 后导致后续无法按平台创建单删除或统计。
  - 平台创建单删除只允许作用于 `source === 'platform'` 的本地平台任务；禅道同步任务、Bug 队列、源禅道任务、父任务、需求和模块等外部字段一律不得被删除或改写。
  - 删除平台创建单不得自动删除该单已产生的 `runs` 执行记录、AI档案、执行日志、工作区和产物证据；这些数据必须继续通过既有“删除执行记录”或“AI档案范围删除”入口独立清理。
  - 删除平台创建单成功后必须广播 `tasks.changed` 事件；负责人和组员端收到后只轻量刷新任务中心数据并先从当前内存/缓存移除被删单，不得触发禅道同步、库存扫描、Git 拉取、本地路径扫描或共享盘扫描。
  - 需要全量扫描时必须由负责人明确要求或另设独立入口。
  - 绿色完成提示必须以真实数据回填为准；同步刚触发时只能显示同步中或已开始同步。
- 禅道任务同步策略：
  - 日常 `同步任务` 必须走部门 `27` 成员当前指派任务的轻量同步，不得默认扫描执行列表再过滤美术人员名下任务。
  - 当前禅道 REST 人员任务查询不可靠时，允许使用禅道经典人员任务页只读发现任务号，例如 `index.php?m=user&f=task&userID=<数字用户ID>&type=assignedTo`；该页面只可作为任务号候选来源，不得信任页面字段作为最终任务事实。
  - 轻量同步必须先读取部门 `27` 用户列表，保留账号、姓名、部门 ID 和禅道数字 `userID`；缺少数字 `userID` 的人员不得用全局执行扫描兜底伪装成快速同步。
  - 部门 `27` 成员数字 `userID` 已验证为：`zhangqw=15`、`fengshuqi=179`、`yushengwei=154`、`yejunbo=151`、`huangjianrong=133`、`lilh=22`、`zhangzb=20`、`lanhj=17`。当禅道 REST 用户列表短暂失败但账号集合仍需 fallback 时，必须保留这些数字 ID 作为经典人员页发现兜底，不得降级成只有账号和姓名的无 ID 人员列表。
  - 轻量同步候选任务号必须包含工作台本地已有、`source === 'zentao'`、存在 `taskNo` 的已跟踪禅道任务号；部门 `27` 成员经典人员任务页发现的当前指派任务号；受时效限制的旧 art-dashboard seen 候选；以及 AI 全流程表中仍在跟踪的任务号。普通 `同步任务` 不得只刷新 `isCurrent !== false` 的任务，否则已被误归档的禅道任务会永远无法恢复。
  - 普通 `同步任务` 的详情刷新范围默认必须使用 `tracked` 或同等语义，刷新当前任务和已跟踪禅道任务；不得在前端或后端 quick 模式里固定传 `detailRefreshScope: "current"` 把同步范围收窄到当前缓存集合。
  - 普通 `同步任务` 如使用旧 art-dashboard 的 `zentao_seen_items*.json`、`zentao_owner_seen_items*.json` 或历史快照，只能作为受时效限制的详情补刷候选；超过时效的旧 seen 记录不得参与普通快速同步，也不得把历史任务重新带回当前列表。
  - 当前任务增量刷新应按任务号逐个读取最新详情，例如 `GET /api.php/v1/tasks/:id`，并与本地缓存字段指纹对比；最终任务状态、标题、指派人、工时、需求、父子任务等必须以详情接口为准。
  - 字段指纹至少覆盖任务标题、状态、指派人、截止日、描述、预计工时、剩余工时、消耗工时、进度、执行、需求、父任务、模块、优先级、更新时间等会影响负责人判断的字段。
  - 禅道任务状态 `pause` 必须在工作台显示为 `已暂停`，不得显示成 `未开始`；同步详情已返回 `pause` 时必须覆盖本地旧状态。
  - 已在工作台跟踪的禅道任务，只要详情接口刷新成功，必须以禅道最新详情为准更新标题、状态、指派人、工时、需求和描述；只有任务未完成、未关闭、未取消且最新 `assignedTo` 仍属于部门 `27` 成员时，才允许保留为当前任务。不得因为本地旧 `isCurrent` 为 true 就把已完成、已转派给外部门或已不在美术当前范围内的任务继续保留为当前。
  - 只有字段指纹变化时才更新本地任务缓存；没有变化的任务不得反复覆盖更新时间或制造无意义变更。
  - 如果任务已完成、关闭、取消或不再属于当前工作范围，必须标记为非当前或归档，不能继续混在当前任务列表。
  - 普通 `同步任务` 一旦通过部门成员经典人员任务页、任务详情刷新或执行任务列表拿到可靠的本次禅道当前结果，必须以本次结果作为当前任务集合；旧的 `isCurrent` 只能在本次禅道当前结果不可用或明显失败时临时保留，不能因“本次结果数量少于旧数据”就把旧当前任务全部补回。
  - 普通 `同步任务` 如果本次详情刷新、人员页发现和执行扫描都没有可靠结果，必须明确保留现有当前任务集合，并在同步摘要中标记保留旧数据；不得向前端返回 `taskCount: 0` 或用空结果触发组员端把当前任务列表刷新成空，也不得弹出绿色“同步完成”误导负责人。
  - 普通 `同步任务` 的绿色完成提示必须以可靠当前快照为前提：人员页发现成功、详情刷新成功且本次 current 集合已按禅道详情回填后，才允许提示完成。若出现 `Unauthorized`、`missing-user-id`、详情刷新全失败或 `currentSnapshotReliable=false`，必须显示保留旧数据/同步未对齐的警告。
  - 同一禅道任务出现重复本地记录时，`isCurrent` 和 `syncStatus` 必须以较新的 `lastSyncedAt/updatedAt/createdAt` 记录为准；不得用 `previous.isCurrent || next.isCurrent` 这类 OR 合并方式把旧 current 状态传染给本次已归档或非当前记录。
  - 当前任务轻量同步可以发现部门 `27` 成员身上新挂的当前任务，但不得同步 Bug，不得扫描全量历史任务，不得扫描全部执行列表。
  - `同步 Bug` 必须走独立 Bug 队列；普通 `同步 Bug` 可以按配置产品做 Bug 队列轻量扫描以发现美术相关未关闭 Bug，同时刷新已跟踪 Bug 详情，但不得触发任务同步或任务全量扫描。
  - Bug 大厅同步范围必须包含未关闭且任一关键人员属于部门 `27` 美术组员的 Bug：`assignedTo` 指派人、`openedBy` 创建人、`resolvedBy` 解决人；不得只按当前指派人过滤。
  - 创建人是美术组员的未关闭 Bug，即使当前指派给外部门或待别人处理，也必须显示在 Bug 大厅，方便负责人追踪闭环。
  - 负责人点击 `同步 Bug` 时必须允许发现上述范围内的新 Bug；不得只刷新已跟踪 Bug 后就结束。
  - `发现新任务/全量扫描` 必须作为独立负责人操作，只在负责人明确触发时扫描执行列表、发现历史执行、补历史候选或做深度兜底。
  - `发现新任务/全量扫描` 可以使用较慢的执行列表或产品 Bug 扫描逻辑，但必须有独立加载态、独立完成提示、独立失败提示和独立权限，不得伪装成普通 `同步任务` 或 `同步 Bug`。
- 任务中心拆单和创单：
  - 任务中心拆单写入禅道时必须逐条返回主单指派和子单创建结果；单条失败不得让其它已成功写入的结果被前端误判为全部失败。
  - 拆单结果必须区分成功数量和失败数量；存在失败时前端必须提示负责人查看失败项或操作日志，不得显示“全部写入成功”。
  - 拆单写入顺序必须先创建并校验子单，再执行主单指派；如果任一子单创建失败，必须跳过主单指派，避免出现“给负责人的单已被指派出去，但子单未创建”的部分成功状态。
  - 拆单创建子单默认只能在原产品下给负责人的美术制作单标题尾部增加负责人确认的后缀，例如 `-2`、`-cocos`、`-15`；除指派目标和负责人明确确认的截止/工时/单独描述外，其它字段必须跟随原本产品下给负责人的美术制作单/制作单模板。
  - 拆单创建子单必须保留父任务、需求、执行、模块、截止日、指派人、任务类型、状态、预计工时、剩余工时、优先级等必要上下文字段；未经负责人明确要求，不得顺手改源禅道任务的标题、描述、工时、状态、优先级或需求关联。
  - 拖拽指派和拆单都不得取消父任务、需求、执行、模块等禅道原有关联；如果表单缺少这些原关联字段，必须停止提交并提示失败，不得用 `0`、空字符串或默认值覆盖禅道。
  - 指派动作只允许提交指派表单和必要备注，不得为了变更负责人而提交编辑任务表单；编辑任务表单会触碰父任务和需求关联，除非负责人明确要求编辑这些字段，否则禁用。
  - 拆单创建子单必须优先读取禅道创建页原始表单并在原始表单基础上覆盖子单名称、负责人、父任务、需求、执行、模块、截止日等必要字段；不得手写一份缺字段的创建表单。
  - 禅道创建页可能包含多张表单；拆单创建子单时必须选择同时包含 `execution`、`module`、`story`、`name` 等核心字段的任务创建主表单，并兼容 `assignedTo[]`、`assignedTo` 等不同指派字段名，不得默认取第一张表单。父任务关联如果由创建 URL 的 `taskID` 承载，页面没有 `parent` 字段时不得误判为缺少父任务。
  - 拆单创建子单必须显式把父任务字段提交为原任务 ID；兼容禅道表单里的 `parent`、`parentID`、`parentTask`、`parentTaskID` 等字段名。即使创建 URL 已带 `taskID`，也不得只依赖 URL 参数导致 POST 后父任务为空。
  - 拆单创建子单提交后必须从禅道查回新任务 ID、执行、需求、父任务/同父任务和指派人；只有查回并确认父任务 ID 等于预期原任务/原父任务时才允许返回成功。不得只凭经典页面 POST 没报错就显示子单创建成功。
  - 拆单创建子单查回后如果父任务为空、为 `0`、为 `-1` 或不等于预期父任务，必须优先通过禅道编辑表单只补挂原父任务并重新查回校验；补挂后仍不等于预期父任务才视为失败并提示负责人，不得把该子单标记为创建成功。
  - 拆单创建子单时，子单描述必须默认继承原任务在禅道产品/任务下的完整任务描述内容；包含 HTML、图片、链接和表格等原描述内容，不得只创建空描述子单。只有负责人明确在拆单项里填写了单独描述时，才允许用拆单项描述覆盖继承描述。
  - 拆单创建子单时，禅道标签、颜色标签、优先级标签、插单/急单等标记只跟主单走；平台不得因为拆单自动给子单新增、继承或覆盖这些标签字段。`-2`、`-cocos`、`-15` 这类负责人确认的子单标题后缀属于任务名称内容，不是标签，必须保留在标题里。

- AI产物 Git/路径扫描：
  - 点击 `刷新库存` 必须抓取 Git 最新内容，不只是统计数量变化；本地路径和共享盘必须通过 `展示管理` 的 `扫描文件夹` 抓取最新内容。
  - Git 刷新和文件夹扫描都必须识别新增、修改、删除；内容变化但数量不变也必须识别为变化。
  - 变化判断至少考虑 commit、提交时间、标题、描述、`preview` 正文、README/Skill 内容等有效指纹。
  - Git 扫描若发现产物内容有变更，质量分必须重新计算。

  - 如果负责人删除了本地路径或共享盘里的文件夹内容，并在路径可访问时主动刷新，平台必须删除该文件夹产物关联的扫描信息、展示配置、贡献人覆盖、别名和隐藏配置，不得只做前端隐藏。

  - 已作废或隐藏的产物不计入 AI 评分，但不得删除源文件或共享盘内容。
  - 修改贡献人只改工作台归属覆盖，不改源文件。
- AI产物调用次数：
  - 调用次数来源于成员同步上报数据和平台累计的产物使用统计，不来自页面假数据。

  - 清除原始明细不得清零或减少原本调用次数；新数据来了，只在旧累计调用次数上增加次数。

  - 操作日志进入调用次数前必须先确认该日志属于真实调用、真实执行终态、真实生成/复用或正向验证；通过后再用操作日志 `targetName` 对象字段、归一后的对象名、`metadata`、`after`、嵌套 records/items/artifacts 中的产物名、Skill 名、路径和别名，与当前库存产物名、展示名、路径、当前别名、历史别名匹配。匹配成功时必须增加对应产物 `usageCount`，同时增加同一使用人的 `usagePeople`；若无法识别真实使用人，必须归入 `未识别使用人` 区域，不得丢弃该次真实调用。`未识别使用人` 只用于明细兜底展示和总次数对齐，不参与有效使用人数、版本升级人数或成员覆盖率计算。
  - AI产物清单外层调用次数、明细弹窗调用次数、成员使用次数、版本有效人数和评分依赖必须共用同一套 `usageCounters.usageCount/usagePeople/usageEventKeys`；不得外层按累计桶、内层按短期日志，或列表和明细分别用不同口径导致内外不一致。
  - 单条操作日志如果携带多条 records/items/artifacts，只能按每条子记录的稳定 ID、路径或对象名拆分事件；同一子记录同时命中产物名、对象字段、路径、当前别名和历史别名时只能累计一次。同一条日志里重复出现同一产物、同一子记录、同一事件键时必须去重，不得因多字段命中把调用次数放大。
  - 操作日志里的验证回填对象必须显示真实 Skill/md/规范产物名称，优先从结构化 `artifactLocation`、`artifactName`、`researchName`、`metadata.skillPath/artifactPath` 和嵌入的 `skills/<name>/SKILL.md` 提取；不得把 `AGENTS`、`README`、`MEMORY`、`试用`、`md` 等通用词显示为对象。只有验证回填对象名或对象路径能命中当前库存真实产物身份时，才允许按该对象增加调用次数，并必须用验证记录唯一键去重，避免同一验证记录因保存日志、自动回填日志、对象名和路径多处命中而重复累计。
  - `研究同步助手`、`同步助手`、`系统同步助手`、`art-progress-reporter`、`member-art-reporter`、`art-workbench-sync-reporter` 都是同步代理人，不得作为 AI 产物有效使用人写入 `usagePeople`。操作日志中代理人产生的 `REPORT_ART_PROGRESS` 仍只算研究同步证据；即使正文或对象字段提到 Skill 名，也不得直接按研究沉淀日志增加调用次数。若同一次同步后产生正向验证回填，则只能通过验证回填记录本身按真实对象名/路径入账一次。
  - `/api/usage-counters` 返回的累计桶必须带当前调用次数口径版本，例如 `logicVersion: usage-only-v7-operation-log-targets`；前端只允许恢复同口径版本的浏览器缓存，旧口径本地缓存必须丢弃并重新读取服务端，避免旧 `count` 或旧 `usageCount` 在刷新页面时回流。
  - AI产物清单首屏快照不得作为调用次数和有效占比的事实来源；快照只保留行结构、版本、贡献人和质量分等展示兜底，展示调用次数时必须重新按当前 `usageCounters.usageCount/usagePeople/usageEventKeys` 装饰。无法命中新口径累计桶时只能显示 `0` 或作废横线，不得回退扫描缓存里的 `row.usageCount`、`skill.usageCount`、旧 `displayUsageCount` 或 `count`。
  - 成员调用明细只能在去重后的真实调用总数内分配；同一事件同时出现中文姓名、账号、别名、路径或多个桶时，只能绑定一个主使用人并累计一次，不得因成员字段重复或别名重叠把总次数放大。
  - 执行调用次数写入成功后必须广播 `usage-counters.changed`；客户端只刷新 `/api/usage-counters`，不得触发库存扫描、Git 拉取、本地路径读取或共享盘读取。
  - 负责人新增或修改调用别名后，平台必须用新别名轻量匹配最近成员上报和工作台操作记录，并把命中结果补入 `usage-counters.json`；不得要求负责人刷新库存、重扫 Git、本地路径或共享盘后才生效。
  - 成员上报或操作记录的摘要正文、标题、Skill 名称、路径、产物名、文件名、结构化 `metadata` 字段都必须参与别名命中；例如负责人把别名改成 `IP形象` 后，上报摘要里出现 `IP形象`、`ip形象`、`IP 形象` 等大小写或标点差异写法时，都应归到对应产物调用次数。
  - 别名补计数必须走事件唯一键去重；同一条上报同时被当前别名、历史别名、产物名或路径命中时，只能累计一次。
- AI产物调用别名数据约定：
  - 别名覆盖必须持久化到 `data/skill-version-overrides.json` 中的 `alias:<projectId>:<relativePath>` 记录。
  - 禁止只把别名写入 `owner:<projectId>:<relativePath>`、普通版本覆盖 key 或前端内存；读取历史数据时必须兼容并镜像到正确的 `alias:` 记录。
  - `aliases` 表示当前有效手动别名，用于界面展示和当前命中。
  - `aliasHistory` 表示隐藏历史别名，只用于历史调用次数归属和去重统计。
  - 保存别名后必须广播 `skill-version-overrides.changed` 平台事件；客户端只拉取 `/api/skill-version-overrides` 并在当前内存 `scans` 上打补丁，不得触发库存扫描。
  - 如果别名保存导致调用次数补录，必须额外广播 `usage-counters.changed` 或同等轻量事件；客户端只刷新 `/api/usage-counters`，不得触发库存扫描、Git 拉取、本地路径读取或共享盘读取。
  - 组员端读取别名必须和负责人端读取同一份服务端覆盖数据。
  - `skill-version-overrides.changed` 只刷新覆盖数据，不刷新库存、不读路径、不拉 Git。
- AI评分和有效占比：
  - `盛威` 的产物不计算有效占比。
  - `盛威` 的工作是单线入口图产物，不能依赖其他人使用人数、其他人验证或互验来得分；他的产物价值和版本必须按本人实际有效使用结果计算。
  - `盛威` 的产物版本按本人有效使用次数计算：本人自行使用 `20+` 次即按 `3.0`，本人自行使用 `3+` 次按 `2.0`，否则按 `1.0`；这个版本口径只影响工作台展示、质量分和 AI评分，不要求别人使用或验证。
  - 其他人的产物计算有效占比时，固定扣除管理员本人和 `盛威`，不是扣除当前登录账号对应人员。
  - AI评分的权重必须突出产物价值，普通组员口径里产物价值分上限必须明显高于使用分和执行分；使用和执行只能作为成果落地佐证，不能盖过产物本身质量、复用价值和版本价值。
  - AI评分必须基于当前平台缓存、产物归属、使用次数、验证/研究同步等真实数据计算，不得使用页面初始假数据。
  - AI评分里成员 `产物数量` 的统计口径，必须基于 `AI 看板累计产物` 与 `当前库存中归属该成员且计入评分的有效产物` 的合并去重结果；不得只看库存，不得只看看板，也不得遗漏任一侧已确认的有效产物。
  - 当库存中的英文 slug、路径名、Skill 名、别名或展示名，与 AI 看板里的中文累计产物本质是同一条时，评分汇总必须优先沿用 AI 看板中的中文产物名并按同一产物合并去重；不得把中文名和英文名分别统计为两条。该规则必须对全员统一生效，禁止只对单个成员写特判。
  - `产物分计算说明` 必须保留。
  - 作废产物、隐藏产物必须从 AI 评分中排除；恢复后重新按正常规则计入。
  - AI评分的执行分只统计有实际执行结果的记录；`passed`、`completed`、`success`、`done`、`conditional_pass` 这类已产出结果且需要/可以进入人工确认的执行可计入。
  - 执行台只有完整失败才不得计入 AI评分；`partial_write`、`部分写入`、已检测到 Figma 真实写入证据、已有产出但最终回读/截图验收未闭环、需要人工确认或继续补验收的记录，都不是完整失败，必须允许计入执行分。
  - 待领取、排队、运行中、取消、删除、草稿、非 0 退出码、没有完成时间/结果证据/写入证据且明确失败或阻塞的执行，不得计入执行分。
  - AI评分里的执行技能数必须按稳定 Skill/md/产物 key 去重；同一个 Skill、同一个 md 或同一产物的不同别名、不同路径写法、重复引用、重复调用，只能算同一个技能/产物。
  - 同一技能重复使用不能按“新的不同技能”增加不同技能数量或执行技能覆盖分，但不能直接丢弃；只要有实际使用结果、闭环记录、完成执行或多人主动使用，就必须按使用次数、使用结果和使用覆盖比例适当增加使用分或执行活跃分。
  - AI评分里的当月使用分必须看本月实际有结果的闭环使用记录；只有调用动作但没有结果、反馈、输出、通过/失败结论、产物证据或适用性结论的记录，不得给当月使用分加分。
  - 执行台数据加到 AI评分时必须以实际执行结果为准；只有完成或有条件完成并带结果证据的执行才能加执行分，单纯创建执行记录、Worker 领取、生成目录、写入日志、排队或运行中不得加分。
  - 修复 AI 评分时，只改评分计算和展示相关逻辑，不得连带改动扫描缓存、贡献人覆盖、权限或刷新策略。
  - 浏览器刷新、页面切换、账号切换或恢复工作台状态时，AI评分必须优先展示上一次已计算并持久化的最终评分快照，不得先显示低分或空分再在几秒后跳成最终分。
  - AI评分快照必须服务端持久化，权威文件为平台 `data/ai-member-score-snapshot.json` 或同等后端存储；浏览器 `localStorage` 只能作为快速显示缓存，不得作为唯一来源。
  - 通过 `127.0.0.1`、`192.168.*`、不同浏览器或不同设备访问同一平台时，AI评分必须读取同一份服务端上次评分快照；不得因为访问地址或前端构建变化读取不同本地缓存导致四个数量、分值和排序变化。
  - 没有触发评分依赖刷新时，普通刷新必须保持原本分值和排序。
  - AI评分栏目必须提供明确的 `刷新评分` 按钮；只有负责人点击该按钮时，才允许重新计算新增调用、产物数量、执行、验证和其它评分依赖并更新评分快照。
  - AI评分默认只展示上次保存的分值；构建、切换页面、刷新浏览器、恢复工作台状态、切换看板、接口静默恢复或做其它功能时，不得自动重算评分，不得让分值闪烁、跳变或先空后变。
  - 浏览器停留在 AI部门看板刷新或恢复工作台状态时，只能恢复 AI 评分快照和 AI部门看板 HTML 显示缓存；不得顺带恢复库存缓存、调用次数、验证记录、研究同步、执行记录或其它评分依赖。只有无任何真实 HTML 看板缓存时，才允许轻量读取成员看板 HTML，且该读取不得触发评分重算。
  - AI评分依赖只有在负责人点击 `刷新评分` 后，才允许读取库存缓存、产物内容指纹、贡献人/展示/作废/隐藏/别名覆盖、调用次数、验证/研究同步、执行统计、成员快照等真实影响评分的数据并重新计算。
  - AI评分重新计算必须在 `刷新评分` 按钮触发后后台执行，不能阻塞 AI部门看板页面进入，不能造成页面卡顿，也不能让负责人看到分数从一版跳到另一版的中间态。
  - 点击 `刷新评分` 后，只有新旧评分依赖签名不一致且计算结果有效时才更新评分快照；扫描失败、接口失败、依赖未恢复完整或计算结果为空时必须保留旧评分快照。
  - `aiMemberScoreRows` 这类页面展示数据只能读取评分快照；没有快照时显示空状态和刷新提示，不得即时计算兜底。
  - AI评分缓存 key 应基于真实轻量依赖签名，例如扫描条目数、Skill 数、扫描更新时间、覆盖数据、调用次数、验证/执行统计等；不得因为页面重新渲染、路由切换或 loading 状态变化就改变缓存 key。
  - 评分刷新逻辑不得触发库存扫描、Git 拉取、本地路径读取、共享盘读取或禅道同步。
- 直接执行到 Figma 数据和执行边界：
  - 当前已定只实现第一场景：无需多轮对话，负责人或有权限账号选择一个 Skill/md，填写 Figma 链接，创建直接执行任务。

  - Figma 写入后截图解析必须区分“Figma MCP 截图已生成”和“本机 shell/curl 无法下载临时 figma.com 图片”：后者只代表本机下载环境受限，不得单独作为 Figma 验收失败或本机阻塞。
  - Figma 写入后验收证据必须来自实际执行结果，例如目标节点回读确认、截图工具返回、视觉复核通过、同类复扫清零、`最终回读/截图验收：已完成` 等；Skill/md 规范说明、操作要求、模板文案里的“必须回读/已验证/成功标准”等文本不得作为实际验收证据。

  - 平台服务器只负责任务创建、队列记录、Worker 心跳、日志回传、状态展示和权限审计。

  - 组员推荐重新执行 `复制开机自启` 命令来更新 Worker；该流程必须覆盖同账号旧自启配置并启动最新 Worker。仅使用 `复制手动启动` 时，必须先关闭旧 Worker 终端或旧 worker 进程，避免同账号同电脑同时跑新旧两个 Worker。
  - Windows `复制开机自启` 必须使用当前用户计划任务安装 Worker，不得依赖管理员权限或 Startup 快捷方式；普通组员 PowerShell 可直接执行，重复安装只覆盖同账号同目录的计划任务。

  - 组员本机已经闭环但断网期间未同步的平台数据，必须在负责人电脑恢复原工作台地址可访问后由 Worker 自动补传；补传失败继续保留有限离线队列，不得阻塞组员当前电脑操作。
  - Figma 必须走原生授权和 Figma MCP；组员登录工作台后，也必须在自己的电脑使用自己的 Figma 账号和本机 Figma MCP。

  - Figma 写入完成不等于整条执行完成；整条执行还必须完成写入后的最终回读/截图验收。缺少最终验收证据时，执行明细必须展示为部分写入阻塞，并提示恢复执行人本机 Figma MCP 授权后继续执行。
- Web 端 Codex 对话数据约定：
  - Web 对话入口复用普通执行权限 `run.codex.execute`，只对非 `direct-skill` 执行记录开放；直接执行仍由执行人本机 Worker 领取，不走平台对话启动。
  - Web 对话提交追加沟通和美术执行台 `重新执行` 都必须调用 `/api/runs/:id/retry` 创建新执行，再通过 `/api/runs/:id/start` 启动；`我来继续执行` 才复用当前记录直接调用 `/api/runs/:id/start`。
  - `/api/runs/:id/retry` 请求体必须使用字段白名单，只允许 `title`、`requirement`、`figmaLinks`、`showdocHints`、`targetPage`、`stage`、`codexRequest` 这类追加执行字段覆盖到新执行；后端必须从原记录复制 md/Skill、Skill 内容快照、自定义流程/模板、执行模式、Figma 写入方式等核心上下文，并强制把新执行归当前点击账号；不得允许前端传入任意字段覆盖项目、权限、状态、旧执行人或路径。
  - `codexRequest` 只允许保存 `model`、`reasoningEffort`、`requestStandard`、`source`；`requestStandard` 必须限制长度，不能保存完整聊天历史、大日志、API Key、Figma token 或本机敏感路径。
  - 后端启动普通执行时必须将全局 Codex 配置和 `run.codexRequest` 合并：API Key、Base URL 和 provider 来自全局配置，单次 `model` 和 `reasoningEffort` 可覆盖本次启动参数。
  - `reasoningEffort` 允许值必须受控，例如 `low`、`medium`、`high`、`xhigh`；不得把未校验字符串直接拼入 Codex CLI 参数。
  - Web 对话默认不新增轮询、不新增持续心跳、不保存多轮全量消息流；只在用户展开并提交时创建一条新执行，避免卡顿和数据暴增。
- Worker 数据与接口：
  - Worker 心跳写入 `agent-workers` 数据，至少包含执行人、设备、最近心跳、Codex 是否就绪、Figma MCP 是否就绪、能力标记和检查消息。
  - Worker 登录平台成功后必须先上报一次基础心跳，再执行 Codex/Figma MCP 自检；不得因为本机 Codex 或 Figma MCP 自检卡住而让工作台一直显示“无心跳”。
  - Worker 的 Codex/Figma MCP 自检必须设置超时，默认 15 秒左右；超时只标记对应能力未就绪并写入检查消息，不得阻塞基础心跳上报和页面状态展示。
  - Worker 的 Codex/Figma MCP 自检必须在执行人自己的电脑上执行，例如检查本机 `codex --help` 和 `codex mcp list`；这些检查只决定该 Worker 是否具备领取执行的能力，不得把自检失败扩大成账号离线或工作台不可用。
  - Worker 空闲状态必须低打扰运行：不得弹窗、不得调用会抢焦点的 GUI、不得打开浏览器或 Figma 窗口、不得造成 Dock 闪烁；轮询只能做后台 Node 请求和必要本机命令检查。
  - Worker 默认任务轮询和心跳兜底间隔为 5 分钟；Codex/Figma MCP 本机自检默认间隔为 40 分钟，即 `ART_WORKER_LOCAL_CHECK_INTERVAL_MS=2400000`。除非负责人明确要求，复制命令、开机自启脚本和 Worker 默认值都必须保持该低打扰配置。
  - Worker 必须在真正领取任务、恢复本机未完成任务或执行结束时才启动较重的 Codex 子进程；空轮询不得每轮重复启动 Codex/Figma 检查进程。
  - Worker 设备花名保存为独立展示字段；后续心跳不得因未携带花名而清空已保存花名。
  - Worker 设备花名只能由设备所属账号本人修改，不允许组员修改他人设备花名。
  - 直接执行领取必须校验执行人账号、项目权限、被分配关系、本机能力和 Worker 权限。
  - Worker 只能领取并回传分配给自己的直接执行任务；除 admin 审计能力外，不得回传他人任务状态。
  - Worker 日志回传要限制单次长度，避免日志过大拖慢平台。
  - Worker 本机状态目录只允许保存当前执行所需的轻量快照、有限离线事件和必要同步去重信息；不得保存整个平台数据、全量日志历史、全量聊天记录、Figma token 或其它长期敏感数据。
  - 工作台前端不得整段渲染原始 Codex/Worker/Figma 日志；`run.log` 中的 Figma 截图、base64、图片、超长 JSON 和大段 MCP 工具返回必须摘要、截断或过滤后再展示。
  - `/api/runs/:id/log` 默认只返回尾部日志摘要；完整原始日志只作为服务器文件保留，排查时查看文件，不得作为页面切换或刷新时的默认渲染数据。
  - 前端默认请求执行日志尾部不得超过 `48KB`；如需排查更大原始日志，只能在明确调试入口或服务器文件中查看，不能作为页面默认加载内容。
  - 美术执行台的历史原始执行日志默认收起；选择已完成/失败/取消的执行记录时不得自动读取历史日志，只有任务运行中或用户点击日志入口时才读取尾部摘要并渲染。
  - 美术执行台的原始执行日志入口固定放在右侧详情顶部操作区 `删除` 按钮右侧，表现为和全局 `操作日志` 一致的图标按钮；不得再放在详情底部折叠块里。
  - 美术执行台点击原始执行日志图标后，必须从右侧半屏 `ElDrawer` 滑出查看内容；抽屉必须直角，内容区域独立滚动，只渲染安全摘要和尾部日志。
  - 美术执行台的原始执行日志收起状态不得提前执行 Markdown 渲染，只有用户点击日志入口打开抽屉时才渲染安全摘要，避免切换页面和刷新卡住。
  - Worker 心跳不得高频写入数据：默认兜底任务轮询间隔为 5 分钟，默认最近心跳刷新间隔为 5 分钟，前端在线展示阈值为 40 分钟；空轮询无任务时不得反复写入 `agent-workers` 状态文件。
  - 美术执行台新建或启动本机 Worker 执行单时，必须保存当前操作账号的 `assignedToUserId`、`queuedForUserId`、`queuedAt`、`executionHost=local-worker` 和 `workerExecution=true`，确保“谁点启动就由谁电脑领取”可被 Worker 按账号准确匹配。
  - 本机执行状态页的待领取/执行中/已完成统计必须覆盖所有本机 Worker 执行单，包含 `direct-skill`、`executionHost=local-worker`、`workerExecution=true` 和美术执行台 `single-skill/custom-workflow` 创建的本机执行记录；不得只统计直接执行记录导致负责人误判。
  - 如果某条执行单的排队时间晚于该执行人 Worker 最近心跳时间，但该 Worker 仍处于 40 分钟在线窗口且 Codex/Figma MCP 已就绪，前端必须按实时事件唤醒口径显示 `正在启动本机执行`；不得把正常在线就绪的任务显示为 `等待下一次心跳/轮询`。只有实时事件连接断开、Worker 离线或自检未就绪时，才把 5 分钟任务轮询作为兜底原因展示。
  - 平台创建或更新本机 Worker 执行单时，必须通过带 `targetUserId` 和 `wakeWorker=true` 的实时事件只唤醒对应执行人的 Worker 立即检查任务；`targetUserId` 只能是当前点击账号，不得使用原执行人或负责人账号兜底；不得广播唤醒所有组员 Worker。
  - 同一条本机 Worker 执行记录被 `继续执行` 或 `我来继续执行` 重新排队时，平台必须清空上一轮执行证据字段，包括 `claimedByDeviceId`、`claimedAt`、`startedAt`、`finishedAt`、`completedAt`、`workerResult`、`workerLocalLogPath`、`workerLocalLogSize`、`logPath`、`promptPath`、`artifactRoot`、`stages`、`blocker`、`resultSummary`、`figmaWriteResult`、`exitCode`、`pid`、`localWorkerStale`、`workerEventIds` 等；不得让 `queued/claimed/running` 因旧证据被误判为本机回传异常。

  - Worker 本机 `state/runs/<runId>/state.json` 只能恢复排队时间之后启动且未完成的同一次本机执行；若平台重新排队时间晚于本机旧 `startedAt`，Worker 必须清空旧 state 的 `startedAt/finishedAt/duration/events/syncedEventIds` 后作为新一轮执行开始，不得复用上一轮开始时间、阶段耗时或离线事件。
  - 最终状态回传为 `completed/failed/blocked/cancelled` 时，平台必须清理旧的 `localWorkerStale/localWorkerStaleDetectedAt` 标记；不得让已经完成且有 Figma 写入和最终验收证据的记录继续显示 `本机回传异常`、`本机阻塞` 或其它旧异常状态。
  - Worker 收到本账号任务唤醒事件后，必须走轻量领取路径：复用最近一次 Codex/Figma MCP 自检结果，立即上报心跳并调用领取接口；不得每次唤醒都强制启动 Codex/Figma 自检进程。5 分钟任务轮询只作为实时事件断线、网络抖动或服务重启后的兜底。
  - Worker 启动后必须先上报一次准备状态；后续只在定时心跳、实际领取任务或执行结束后刷新 Worker 状态。
  - Worker 不能依赖组员浏览器网页是否置顶或是否正在操作工作台；只要组员电脑启动 Worker 且登录成功，就必须能接收直接执行任务。
  - 工作台前端的 Worker 状态自动刷新必须克制：只有浏览器标签页可见，且当前在美术执行台或本机执行状态页时，才因 `agent-workers.changed` 事件刷新 Worker 状态。
  - `agent-workers.changed`、`runs.changed`、任务唤醒和其它后台实时事件触发的 `runs` / `agentWorkers` 刷新必须使用静默刷新和并发去重；不得设置人工按钮 loading，不得连续闪烁、卡住页面或提高 Worker 轮询/本机自检频率。

  - 涉及负责人管理、组员执行、全员查看、本人维护、批量删除、刷新扫描、同步禅道、复制命令、创建执行、Worker 回传、AI档案删除等不同风险级别的操作，必须拆成不同权限项。
  - 新增权限时必须同时检查 `server/auth.mjs` 的权限目录、默认角色权限、历史角色兼容补丁，`src/App.vue` 或对应视图的 `can(...)` 判断，以及 `RoleManagementView.vue` 的展示是否能配置。
  - 负责人/admin 默认拥有管理类权限；组员默认只拥有必要读取、明细查看和被授权的执行能力。
  - 能查看页面不等于能编辑、删除、刷新、作废、恢复、复制命令或批量清理。
  - 所有写接口必须有后端权限校验；前端隐藏按钮只是交互辅助，不得作为唯一权限控制。
  - 与任务中心禅道同步相关的权限必须拆分；现有粗粒度 `task.sync` 只能作为历史兼容，不得作为新增同步能力的唯一权限：
    - `task.sync`：普通任务增量同步按钮。
    - `bug.sync`：普通 Bug 同步按钮。
    - `task.discovery.sync`：发现新任务或全量扫描按钮。
    - `api.tasks.sync`：普通任务增量同步 API。
    - `api.bugs.sync`：Bug 同步 API。
    - `api.tasks.discovery`：发现新任务或全量扫描 API。
    - `task.platform.delete`：删除平台创建任务按钮。
    - `api.tasks.deletePlatform`：删除平台创建任务 API。
  - 新增或拆分任务同步、Bug 同步、发现新任务、全量扫描接口时，必须同步更新 `server/auth.mjs` 权限目录、默认角色权限、历史权限兼容、前端 `can(...)` 判断、角色管理页面展示和后端 `requireProjectAccess`/权限校验。

    - `menu.agentWorkers`：查看本机执行状态页。
    - `menu.aiArchive`：查看 AI档案页。
    - `run.directSkill.create`：创建直接执行。
    - `run.directSkill.workerCommand`：复制 Worker 手动启动或开机自启命令。

    - `api.agentWorkers.read`：读取 Worker 状态 API。
    - `api.agentWorkers.heartbeat`：Worker 心跳 API。
    - `api.agentWorkers.alias`：本人修改 Worker 设备花名 API。
    - `api.agentRuns.claim`：Worker 领取直接执行 API。
    - `api.agentRuns.log`：Worker 回传日志 API。
    - `api.agentRuns.status`：Worker 回传状态 API。
    - `api.aiArchive.delete`：AI档案范围删除 API。
  - 与 AI部门看板和评分相关的权限必须拆分：
    - `menu.aiMembers`：进入 AI部门看板一级页面。
    - `menu.aiMembers.owner`、`menu.aiMembers.member`：历史权限字段，仅用于兼容旧角色数据；新功能不得依赖它们控制 AI部门看板下方 HTML 内容。
    - `aiMembers.score.view`：查看当月 AI 评分。
    - `aiMembers.score.refresh`：点击 `刷新评分` 并手动重算 AI 评分快照；该权限默认只给工作台管理者或明确授权角色。
    - `api.aiMembers.read`：读取 AI部门成员快照和看板 HTML 缓存。
    - `api.aiMembers.score.read`：读取 AI 评分所需轻量缓存或快照。
    - `api.aiMembers.score.write`：保存负责人手动刷新后的 AI 评分快照；默认只给工作台管理者或明确授权角色。
    - `api.aiMembers.refresh`：刷新 AI部门成员快照或研究同步；不得等同于刷新库存。
  - `menu.runs` 只代表美术执行台页面；`menu.agentWorkers` 只代表本机执行状态页面；`menu.aiArchive` 只代表 AI档案完整明细页面，三者不得混用。
  - `api.runs.execute` 保留给普通执行任务创建、启动、中断等旧执行能力；直接执行创建和 Worker 流程必须使用专门权限。
  - 能复制 Worker 命令不等于能管理账号；全员账号、展示密码等敏感数据仍应受账号管理权限控制。
  - 与 AI产物清单相关的权限必须保持拆分：`skill.scan.refresh`、`skill.source.connect`、`skill.source.edit`、`skill.source.delete`、`skill.asset.create`、`skill.asset.void`、`skill.assetOwner.manage`、`skill.version.manage`、`skill.alias.manage`、`skill.usageLogs.view` 及对应 `api.*` 权限不得互相替代。

  - `/api/skill-version`：保存版本、展示名称、贡献人、类型或带版本的别名变更；必须按具体变更校验相应权限。

  - `/api/usage-counters`：读取平台累计调用次数事实缓存；`usage-counters.changed` 事件只允许触发该接口轻量刷新。
  - `/api/tasks/sync-zentao`：普通任务增量同步入口；`syncKind: "task"` 时只能执行部门 `27` 成员当前指派任务的轻量发现和详情刷新，不得同步 Bug 或执行全量发现。
  - `/api/bugs/sync-zentao`：普通 Bug 同步入口；只同步 Bug 队列，不得触发任务同步。

  - 发现新任务或全量扫描必须使用独立接口或独立 `syncProfile/mode`，并接受独立权限校验；不得让普通 `同步任务` 请求默认进入全量扫描。
  - `/api/runs`：创建普通执行；当 `sourceType` 或 `executionMode` 为 `direct-skill` 时创建直接执行。
  - `/api/runs` 创建普通执行时必须支持无任务关联的独立执行：无 `taskId` 且未显式要求创建任务时，只写 `runs` 执行记录和产物目录，不写 `tasks` 任务中心数据。
  - `/api/runs/:id/retry`：用于 Web Codex 对话追加沟通和美术执行台 `重新执行`，基于当前执行记录创建一条新的执行记录；只允许白名单字段和 `codexRequest` 覆盖新执行，核心 Figma 链接、md/Skill、Skill 快照、流程/模板和执行要求必须从原记录继承，旧运行状态和结果证据不得继承。
  - `/api/runs/:id/start`：启动普通执行；必须应用全局 Codex 配置和该执行上的 `codexRequest` 单次模型/推理配置；不得启动 `direct-skill`。`start/resume` 只能把当前 `run.id` 排队给当前点击账号本机 Worker；请求体中的 `targetUserId` 不得被用于改派给原执行人或他人。`mode: "resume"` 表示业务断点续跑，必须保留旧上下文和可续跑线索；`mode: "restart"` 只能用于已经由 `/retry` 新建出的记录从第一步执行，不得清空原记录。
  - `/api/runs` 返回的执行记录必须包含阶段耗时相关字段：`startedAt`、`finishedAt`、`durationMs`、`stages[].startedAt`、`stages[].finishedAt`、`stages[].durationMs`；历史补齐出来的估算耗时必须标记 `durationEstimated: true`，不得伪装成原始实时计时。
  - `scripts/restore_run_durations.mjs` 是执行记录耗时恢复脚本；运行前必须先备份 `data/runs.json` 到 `data/restore-backups/`。该脚本只允许补齐 `durationMs`、`startedAt`、`finishedAt`、`durationEstimated` 和必要阶段状态，不得修改 Figma 链接、补充要求、任务描述、Skill/md 线索、执行对象、执行模式、执行人、设备、日志正文或其它业务上下文字段。
  - `DELETE /api/runs?from=...&to=...`：AI档案按筛选范围删除执行明细；只允许有 `api.aiArchive.delete` 权限的账号调用。
  - `/api/runs/direct-skill`：创建直接执行。
  - `/api/agent-workers`：读取 Worker 状态。
  - `/api/agent-workers/heartbeat`：Worker 心跳。
  - `PATCH /api/agent-workers/:id/alias`：本人修改 Worker 设备花名。
  - `/api/agent-runs/next`：Worker 自动领取直接执行。
  - `/api/agent-runs/recover`：Worker 只按执行人本机已有未完成快照恢复同设备直接执行；不得无本机快照恢复旧 `running` 记录。

  - `/api/agent-runs/:id/sync`：Worker 批量补传离线日志、状态和阶段耗时事件；必须按 `runId + eventId` 幂等更新原执行记录。
  - `/api/agent-runs/missing`：Worker 上报本机仍保留的 `runId` 列表，平台只返回已经不存在的执行记录 ID，用于执行人本机清理缓存；不得作为长期删除墓碑写入无限增长数据。
  - `/api/agent-runs/:id/log`：Worker 回传执行日志。
  - `/api/agent-runs/:id/status`：Worker 回传执行状态。
- Git 自动备份数据约定：
  - 自动备份只提交 Git 工作区内未忽略的项目文件。
  - `.gitignore` 中的 `data/`、`logs/`、`dist/`、`node_modules/`、`.env*`、`outputs/`、`workspace/` 等不得被自动备份提交。

  - 自动备份不做强推、不做空提交。
  - 自动备份提交信息使用中文时间戳格式：`自动备份 YYYY-MM-DD HH:mm`。

## 4.禁用写法、已定功能、固定执行命令

- 禁止事项：
  - 禁止修改负责人未要求处理的模块、字段、样式、交互、脚本、接口、数据和流程。
  - 禁止把已确认保持原样的规则、弹窗、流程、字段或布局借其它功能顺手调整。
  - 禁止未经要求重构业务结构。
  - 禁止顺手修改禅道字段。
  - 禁止为了实现新功能引入会导致页面切换卡顿、内容消失、列表闪烁、长期 loading 或数据无缘由累加的重逻辑。
  - 禁止平台事件在非当前页面直接触发大刷新；非当前页必须先标记待更新，进入页面后再按需轻量补数据。
  - 禁止同一类自动刷新在短时间内重复并发请求；禁止用后台冷却拦截负责人主动点击后的真实刷新。
  - 禁止在列表行渲染、步骤渲染、成员卡片渲染或档案明细渲染中反复全量扫描 `runs`、`tasks`、`taskReviews`、`agentWorkers` 等数组；必须改用索引、分组、分页或虚拟化。
  - 禁止扫描失败、路径断联、网络失败或接口异常后用空数据覆盖缓存。
  - 禁止页面进入、普通刷新、打开弹窗或修改工作台配置时自动重新抓取库存。
  - 禁止在负责人未点击对应刷新按钮、源数据无变化、依赖无变化时重新计算分数、刷新文件扫描、重建统计或覆盖缓存。
  - 禁止刷新页面后把 AI产物清单置空或要求重新接入扫描。
  - 禁止刷新任务中心时先把任务、Bug、验收统计显示为 0，再等待接口或缓存恢复后跳回上次数据。
  - 禁止 AI产物清单刷新后只显示统计卡、不显示上次首屏清单，除非平台从未生成过任何真实清单快照。
  - 禁止把 AI产物清单首屏展示快照当作真实扫描缓存、评分依据、调用次数依据或版本/贡献人/别名覆盖依据。
  - 禁止把 AI产物刷新做成只看数量变化。
  - 禁止组员端因为无扫描权限而看不到平台缓存。
  - 禁止作废产物或隐藏产物继续计入 AI评分。
  - 禁止用别人使用人数、别人验证或互验作为 `盛威` 单线入口图产物的主要得分依据。
  - 禁止 `盛威` 产物自行使用 `20+` 次后仍因缺少多人使用或互验而停留在 `1.0/2.0`。
  - 禁止把未实际完成、失败、阻塞、取消、排队、运行中或非 0 退出码的执行计入 AI评分执行分。
  - 禁止把同一个 Skill/md/产物的不同别名、不同路径写法或重复引用当成多个不同技能给 AI评分加分。
  - 禁止把 AI 看板中的中文累计产物和库存中的英文 slug/路径名/Skill 名当成两条不同产物分别计入成员 `产物数量`；同物必须先归一再统计。
  - 禁止因为同一 Skill/md/产物已经按稳定 key 去重，就把重复使用产生的实际成果、闭环次数、完成执行次数或多人使用比例全部丢掉不计分。
  - 禁止浏览器刷新、页面切换或账号切换后重新计算 AI评分并展示可见跳分。
  - 禁止构建、切换页面、恢复工作台状态、打开 AI部门看板或接口静默恢复时自动刷新 AI评分；未点击 `刷新评分` 时必须保持上次分值。
  - 禁止在评分依赖未刷新时重算 AI评分并覆盖原本分值。
  - 禁止 AI评分后台重算失败或依赖数据未恢复完整时用空分、低分或临时分覆盖已有最终评分快照。
  - 禁止 AI评分刷新触发库存扫描、Git 拉取、本地路径读取、共享盘读取或禅道同步。
  - 禁止 AI部门看板下方 HTML 看板因为外层 ready/loading 标记未恢复而长期显示“正在加载”或空白。
  - 禁止有缓存 HTML 时把 AI部门看板 iframe 隐藏到 loading 占位之后。
  - 禁止把 `正在加载 AI部门看板...` 这类兜底 HTML 保存成 AI部门看板正式缓存，或用它覆盖最后一次真实 HTML。
  - 禁止 AI部门看板刷新后出现先一版内容、几秒后跳成另一版内容的可见中间态。
  - 禁止删除源文件或共享盘内容来实现作废、隐藏或工作台展示调整。
  - 禁止共享盘或本地路径断联时清空已扫描产物。
  - 禁止把接入扫描弹窗做成实时刷新库存入口。
  - 禁止本地路径或共享盘内容删除后只做前端隐藏而不清理扫描关联信息。
  - 禁止成员上报原始明细清理时清零累计调用次数。
  - 禁止删除操作日志后回退、扣减、清零或重算已经累计的 AI 产物调用次数。
  - 禁止删除操作日志后再反写新的删除审计日志，导致“删除全部日志”后仍留存一条操作日志。
  - 禁止独立页面的删除按钮越权删除其它页面或其它业务入口的数据；例如操作日志删除不得影响美术执行台或 AI档案执行记录。
  - 禁止把 `data/runs.json`、任务中心数据、Bug 队列、人工验收、AI档案等业务事实数据加入短期 retention 清理；短期 retention 只能处理已定义的原始日志、原始事件或短期明细缓存。
  - 禁止操作日志删除接口清理 `runs`、任务中心、Bug、验收、Worker、AI产物清单或其它有独立删除入口的业务数据。
  - 禁止用阶段报告模板里的 `未执行`、`待执行`、空耗时或默认步骤覆盖 `runs.json` 中已有的真实执行状态、阶段状态、起止时间和耗时。
  - 禁止恢复或补齐执行记录耗时时把 `finishedAt` 写成早于 `startedAt` 的时间；无法可靠判断结束时间时必须保守保留原值或标记估算。
  - 禁止把历史估算耗时当成精准实时计时；估算值必须标记 `durationEstimated: true`，界面和后续逻辑不得据此覆盖真实计时。
  - 禁止因为负责人修改产物别名而丢失旧累计调用次数。
  - 禁止把一次调用因命中多个别名或名称重复计数。
  - 禁止调用别名保存到 `owner:` 覆盖 key 后就视为成功；别名必须落到 `alias:` 覆盖 key。
  - 禁止只在当前浏览器内存里更新别名而不写服务端覆盖数据。
  - 禁止通过刷新库存、重扫 Git、读取共享盘或读取本地路径来实现别名实时同步。
  - 禁止负责人改了调用别名后，只保存别名而不对已有上报/操作记录做轻量补计数。
  - 禁止只匹配结构化字段而忽略成员上报摘要正文里的调用别名。
  - 禁止把 `usage-counters.changed` 处理成库存刷新、Git 拉取、本地路径扫描或共享盘扫描。
  - 禁止有效占比按当前登录账号动态扣除；有效占比固定扣除管理员本人和 `盛威`。
