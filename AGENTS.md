# 项目Agent开发基线

## 1.项目信息、技术栈版本

- 项目路径：`/Users/se7en/ArtProject/platform`
- 项目定位：美术部门生产工作台，用于禅道任务/Bug 同步、任务中心、AI产物清单、成员产物、质量评分、库存缓存、美术执行台、本机直接执行状态、AI档案和负责人管理操作。
- 默认回复语言：中文。
- 默认用户身份：美术部门负责人。涉及任务、排期、周报、资源协调、风险识别、权限配置、组员管理、Worker 状态、AI产物清单和执行状态时，优先按工作台美术负责人管理者视角组织回复。
- 术语边界：
  - `admin`、`美术负责人`、`负责人管理者`、`工作台管理者` 指工作台内具备管理权限的美术负责人账号。
  - `执行人`、`组员`、`被指派人` 指实际领取并执行任务的人。
  - `任务负责人`、禅道原始负责人不得与工作台 `admin` 混用。
  - 用户说“我”“负责人”“美术负责人”“负责人管理者”“工作台管理者”时，默认指工作台里的美术负责人管理者 `admin`，不是某条任务的执行人。
  - 用户说“基线”时，默认按本文件要求更新 `AGENTS.md`：只保留已确认硬性约束、功能边界、数据结构、接口约定、权限约定、禁用写法和固定命令；不得写入调试代码、临时方案、作废方案或闲聊内容。
  - 用户要求“汇总本次全部会话内容”“输出标准 AGENTS.md 全文”“后续新会话自动加载遵守”时，必须直接更新本文件；若本文件已存在，必须结合旧规则和本次新增规则提炼，不得简单覆盖或删除已确认硬约束。
  - 当前会话内容较多、接近上下文压缩、或已经形成稳定的新界面/新操作/新权限/新数据约定时，即使用户没有再次完整复述，也应主动按“基线”口径更新本文件。
- 用户可见文案、说明文档、标记文档、看板文案统一使用中文。代码标识、命令、路径、文件名、字段名、接口参数、第三方专有名词可保留英文。
- 前端技术栈：
  - Vue 3
  - Vite 5
  - Element Plus
  - Sass
  - Node.js `>=20`
- 后端技术栈：
  - Node.js ESM
  - `mysql2`
  - 本地 JSON/缓存文件
  - macOS LaunchAgent
- Git 远端：
  - `origin`：`https://github.com/se7en905/ArtPlatform.git`
  - `platform-backup`：`https://github.com/se7en905/gongzuotgi.git`
- Git 自动备份：
  - 由本机 macOS LaunchAgent 定时执行，不经过 Codex 对话，不调用 AI，不消耗 Codex/OpenAI token。
  - 默认每 30 分钟检查一次，有变更才自动提交并推送。
  - 自动备份远端固定为 `platform-backup/main`。
  - 自动备份脚本必须只依赖本机 Git/GitHub 凭据，运行时不得使用 AI、Codex 或 OpenAI 环境变量。

## 2.命名、组件、样式、目录规范

- 保持现有目录、模块边界和已确认交互流程；不得为了局部修复顺手重构、改名、换布局或调整业务结构。
- 关键文件职责：
  - `server/server.mjs`：平台接口、缓存读取、禅道同步入口、权限判断、本机 Worker 状态和直接执行任务接口。
  - `server/store.mjs`：本地 JSON 数据读写、执行记录、Worker 心跳、直接执行领取和状态回传。
  - `server/auth.mjs`：账号、角色、权限目录、默认角色和历史权限兼容。
  - `server/scanner.mjs`：Git、本地路径、共享盘扫描与产物内容解析。
  - `src/App.vue`：全局业务状态、路由守卫、权限判断、库存缓存读取、刷新库存、贡献人覆盖、AI评分、产物使用统计、直接执行创建和 Worker 状态读取。
  - `src/views/SkillInventoryView.vue`：AI产物清单、贡献人调整、使用明细抽屉、成员产物整合入口。
  - `src/components/dialogs/SkillPreviewDialog.vue`：Skill/md 内容预览、调用别名维护和直接执行到 Figma 的创建入口。
  - `src/components/dialogs/RunCreateDialog.vue`：新建美术执行弹窗。
  - `src/views/RunsView.vue`：美术执行台、执行详情、直接执行的本机状态提示。
  - `src/views/AgentWorkersView.vue`：本机执行状态页，展示 Worker 心跳、自检和执行明细。
  - `src/views/AiArchiveView.vue`：AI档案页，展示完整 AI 执行明细、筛选和范围删除。
  - `src/views/RoleManagementView.vue`：角色管理和细粒度权限配置。
  - `src/styles.scss`：全局展示兜底样式、抽屉、评分面板、弹窗分段控件样式。
  - `scripts/art-direct-worker.mjs`：组员本机直接执行 Worker。
  - `scripts/install_art_direct_worker_launch_agent.sh`：安装组员本机 Worker 开机自启。
  - `scripts/git-auto-backup.sh`：Git 自动备份执行脚本。
  - `scripts/install_git_auto_backup_launch_agent.sh`：Git 自动备份 LaunchAgent 安装脚本。
- 工作台不是营销页或展示页。UI 优先保证数据准确、信息完整、操作稳定、密度合理。
- 工作台关键业务文本不得无意义省略，尤其是按钮、分段控件、标签、抽屉标题、成员名、产物名、版本记录、任务标题和操作字段。
- 长路径、日志、表格大段文本可以受控换行或提供详情入口，但不能让负责人无法判断真实内容。
- 列表操作栏规则：
  - 任务中心操作栏固定在右侧。
  - AI产物清单操作栏固定在右侧。
  - 响应式宽度变化时操作字段不得消失。
  - 操作按钮必须左右横向排列，不得换行导致按钮不可见。
- 全局右侧半屏抽屉：
  - 右侧半屏抽屉必须为直角表现。
  - 所有通过 Element Plus `ElDrawer` 从右侧滑出的界面都必须全局直角，包含抽屉本体、头部和内容区。
  - 不得用 `.el-drawer.rtl`、`.art-progress-log-drawer`、`.skill-usage-drawer` 或类似选择器恢复圆角。
  - 通过 `append-to-body` 渲染时，样式必须同时考虑全局样式和组件内 scoped 样式。
- “新建美术执行”弹窗：
  - 执行来源和执行范围分段控件必须使用全倒角选中底图。
  - 文字必须完整可见。
  - 不得改变已确认交互流程。
- 拖拽规则：
  - 任务拖拽必须稳定。
  - 任务只要在对应成员身上，就必须能拖拽成功。
  - 拖拽稳定性与禅道同步无关，不能因同步状态影响本地拖拽成功率。
- 任务中心和禅道同步入口：
  - 任务中心的 `同步任务` 是日常快速刷新入口，只刷新当前已有禅道任务，不承担发现新任务或全量扫描职责。
  - Bug 大厅的 `同步 Bug` 是独立入口，只同步 Bug 队列，不得顺带触发任务同步。
  - 如需要发现新指派给美术部门的任务，必须另设 `发现新任务`、`全量扫描` 或同等语义的独立入口，文案必须让负责人明确知道该操作可能较慢。
  - `同步任务`、`同步 Bug`、`发现新任务/全量扫描` 在界面、加载态、完成提示和权限控制上必须彼此独立。
- AI产物清单规则：
  - 负责人可见 `明细`、`作废`；已作废后按钮显示 `恢复`。
  - 组员只可见 `明细`。
  - 作废后版本展示为置灰 `1.0`；恢复后恢复正常版本展示。
  - 作废和恢复只影响工作台展示、评分计入和状态，不修改源文件。
  - 本地路径和共享盘扫描产物只显示文件夹名称或负责人配置的展示名称，不显示绿色说明框、绿色角标或额外来源说明。
  - 展示管理必须支持全选、是否展示、展示名称/产物名称、调用别名、贡献人多选、类型选择。
  - 类型选项为 `Skill`、`规范`、`文件夹产物`；类型只影响工作台展示、统计分类和质量分入口，不修改源文件。
- AI产物调用别名界面规则：
  - Skill/md 内容预览里的 `调用别名` 是负责人维护产物命中关系的入口。
  - 别名输入框允许多个别名，使用顿号、逗号或换行分隔。
  - 保存后当前电脑和其他组员都必须能看到同一份新别名。
  - `当前别名` 只显示当前有效手动别名，不显示隐藏历史别名。
  - 没有手动别名时，默认别名优先使用产物名称和文件内容触发词；有手动别名时必须优先使用手动别名，不得被文件触发词覆盖。
  - 修改别名只影响工作台展示、调用次数匹配、质量分统计和命中关系，不修改 Git、本地路径或共享盘源文件。
- 接入扫描弹窗：
  - 弹窗内只负责选择来源并开始扫描，来源为 Git 仓库、本地路径、共享盘路径三选一。
  - 弹窗内不放 `刷新库存` 入口；总界面保留统一 `刷新库存` 入口。
  - 已接入来源、来源名称、上次扫描状态和上次扫描内容默认展示平台缓存，不因打开弹窗而实时刷新。
- 美术执行台和直接执行到 Figma：
  - Skill/md 内容预览里可以展示 `直接执行到 Figma` 区块。
  - 直接执行创建弹窗至少包含执行产物、Figma 链接、写入方式、执行人、补充要求。
  - 创建直接执行后只进入队列，不显示平台侧“开始”按钮。
  - 美术执行台的直接执行详情必须展示执行人、本机 Worker 状态、领取设备、最近心跳和阻塞提示。
  - 执行内容展示只显示具体使用的产物/Skill/md 名称和类型，例如 `界面收尾`、`Skill`；不得在执行内容或类型里展示 `/Users/...` 等本地路径。
  - 美术执行台允许无禅道依据、无任务中心依据的独立执行，用于实验、复查、验证已完成单和 Figma 操作测试；这类执行只需要保留执行过程、结果、日志、产物目录和 AI档案追溯，不得自动在任务中心新增一条“平台创建”任务，也不得默认挂在当前账号或某个组员身上。
  - 只有从任务中心点击 `发起执行`，或新建执行时明确选择了任务中心记录并带有 `taskId`，才允许和任务中心任务关联；手填“已完成任务 ID / 禅道 ID”只能作为上下文线索，不得因此自动创建任务中心记录或自动关联任务中心记录。
  - 新建美术执行弹窗默认来源必须是 `独立执行`；`关联任务` 只在负责人明确选择任务中心记录时使用。
  - 无任务关联执行的 `sourceType` 必须使用 `standalone` 或同等独立来源标识；任务中心发起的执行使用 `task-center` 或明确任务关联标识。前端判断任务链路、人工验收入口、任务中心执行次数时必须先判断是否确实有关联任务，不得仅凭 `zentaoId/taskNo` 字符串匹配。
  - 独立执行创建后只刷新 `/api/runs`，不得刷新任务中心、触发禅道同步、库存扫描、Git 拉取、本地路径扫描或共享盘扫描。
  - 美术执行台右侧详情必须按执行类型拆分展示：普通“新建美术执行”或分阶段执行才展示关键动作概览、任务链路和继续和 Codex 沟通。
  - 引用单个 md/Skill 的直接执行（`direct-skill`）右侧只展示 Worker 状态、轻量阶段进度、直接执行结果、执行对象、执行环境、问题提示和 AI档案入口；不得展示关键动作概览、任务链路、查看具体明细式链路文案或继续和 Codex 沟通入口。
  - 在“新建美术执行”里已选择 md/Skill 的单技能执行（`art-single-skill`）也不得展示关键动作概览；右侧必须按正常操作流程展示执行步骤、结果明细和日志。
  - 单 md/Skill 执行右侧标题说明必须使用 `查看当前任务的执行步骤明细` 口径。
  - 如果任务链路本质是在表达步骤流程，单 md/Skill 执行时必须上移到当前状态下方、阶段进度附近，标题使用 `执行步骤明细`；不得放在关键动作概览下方作为独立下沉模块。
  - 单 md/Skill 执行步骤必须使用横向流程步骤图展示，按 `选择执行内容`、`读取任务上下文`、`执行自检`、`Codex 执行`、`结果回传`、`归档验收` 这类负责人可理解的步骤呈现；当前运行步骤必须高亮，已完成、失败、待执行要有清晰状态。
  - 单 md/Skill 执行结果必须模块化展示交付判定、成功数量、失败数量、扫描点、数据类、执行对象、执行环境、问题提示和 `查看档案` 入口；不得默认重复展示旧的纵向任务链路或关键动作概览。
  - Web 端 Codex 对话入口必须做成 Codex 官方图标悬浮按钮，不得作为右侧详情里的固定卡片或执行过程中的常驻面板。
  - Web 端 Codex 悬浮入口只在用户点击左侧美术执行清单中的某条非 `direct-skill` 执行记录后出现；未选中左侧记录、当前执行正在运行、或记录是 `direct-skill` 时必须隐藏。
  - Web 端 Codex 悬浮入口允许在当前页面拖动位置；拖动位置只作为当前页面临时 UI 状态，不写入后端，不产生额外数据记录，拖动时不得误触发打开对话。
  - 点击 Web 端 Codex 悬浮入口后，打开的对话只对应当前左侧选中的执行记录；提交追加要求必须创建新的执行记录继续处理，不能把要求写到其它任务上。
  - Web 端 Codex 对话窗口必须默认收起；单 md/Skill 执行不允许直接展开对话窗口。展开后才渲染输入框、模型、推理强度和请求标准。
  - Web 端 Codex 对话提交后创建新的执行记录，不在当前记录里追加长对话历史；默认不得读取或渲染历史长日志、整段模型事件或大段上下文。
  - Web 端 Codex 对话可选择模型和推理强度；单次配置只作为新执行的 `codexRequest` 保存和启动参数使用，不得保存 API Key 或 Figma OAuth token。
  - 直接执行如果后端没有阶段数据，右侧阶段进度必须兜底展示 `已创建`、`Worker 领取`、`本机执行`、`结果回传`，让负责人能按步骤判断状态。
  - 分阶段普通任务仍必须按阶段一步步展示，适合使用横向阶段步骤呈现；不得因直接执行轻量视图而删除普通分阶段任务的阶段进度。
- `/agent-workers` 本机执行状态页：
  - 必须作为一级页面展示，不得藏在深层入口。
  - 页面必须区分工作台管理者和组员美术执行者。
  - 管理者可见完整的直接执行流程说明、管理判断文案和给组员分发命令的说明。
  - 非管理者不得看到页面开头“本机执行状态 / 查看组员 Worker 在线...”整条提示、`直接执行启动方式` 模块、管理者口径的 `本机 Worker 绑定与实验操作` 文案。
  - 组员必须能复制自己的 `复制手动启动` 和 `复制开机自启` 命令。
  - `Worker 心跳` 是固定模块名，不得改回 `组员本机 Worker`。
  - `Worker 心跳` 模块标题下只保留“命令必须在执行人自己的电脑运行。”这类操作提示。
  - `Worker 心跳` 模块头部右侧必须放 `复制手动启动`、`复制开机自启`、`刷新状态`，按钮左右横向排列。
  - `Worker 心跳` 模块不得放顶部六个统计小块；在线、就绪、待领取、执行中、已完成等信息必须并入每个组员卡片。
  - `Worker 心跳` 必须按可见组员展示 Worker 信息；没有心跳的可执行组员也要展示；组员账号也能看到全部可见组员卡片。
  - 卡片里的组员名称必须优先使用平台账号展示名绑定 Worker `userId`；不得只显示 `admin`。
  - 卡片内必须展示组员名称、设备信息、Codex 是否就绪、Figma MCP 是否就绪、最近心跳、待领取数量、执行中数量、已完成数量。
  - Worker 设备允许本人维护展示花名；花名只影响工作台展示，不覆盖真实 `deviceId`、真实设备名或心跳绑定关系。
  - Worker 设备展示必须优先使用本人维护的花名；`Worker 心跳` 卡片、美术执行台直接执行详情、执行明细里的领取设备都必须同步展示花名，且所有成员可见同一展示。
  - `Worker 心跳` 的心电图表现以“该组员本机是否可以直接执行”为依据：只有 Worker 在线且 Codex 与 Figma MCP 都可用时，才显示绿色动态心跳；没有心跳或任一能力未就绪时，显示红色直线。
  - `执行明细` 是固定模块名，不得改回 `直接执行队列`。
  - `/agent-workers` 的 `执行明细` 只展示最近 10 条直接执行记录；完整执行明细放在左侧一级菜单 `AI档案`。
- `AI档案` 页面：
  - `AI档案` 是一级菜单，用于展示完整 AI 执行明细。
  - 必须支持关键词、人员、类型、状态、开始时间和结束时间筛选。
  - 筛选项变更后默认立即生效，不需要额外 `查询` 按钮。
  - 关键词搜索框固定放在页面标题卡片右上角。
  - 表格执行内容只展示具体使用的产物/Skill/md 名称和类型，不展示本地路径。
  - 执行内容不使用按钮式底色和描边，只保留可点击文本表现。
  - 页面顶部必须展示当前筛选范围内的 `归档任务`、`已闭环`、`待返工`、`待验收` 四项汇总。
  - 顶部四项汇总必须表现为明显分开的四块区域，并且支持点击筛选；点击后下面执行列表必须精准展示对应类别记录，`归档任务` 表示当前筛选范围内全部记录。
  - 每条执行记录必须能在 AI档案当前页面直接查看明细，不得因点击明细跳回美术执行台或任务中心。
  - 明细必须展示该次执行的成功数量、失败数量、扫描点数量和具体数据类；数据优先来自 `run.resultSummary`、`stages`、`changeSummary`、验证命令、产物证据和 Figma 链接等结构化字段。
  - 执行明细必须按负责人易读方式组织，优先展示交付结论、执行对象、执行环境、问题与待处理、本次检查内容、产物与证据；不得重复展示同一字段，也不得默认展示 `thread.started`、`item.completed` 等模型事件串。
  - AI档案明细默认不得读取或渲染整段原始日志；原始日志只能按需、轻量、尾部读取，避免点击查看档案卡顿。
  - AI档案列表必须分页或虚拟化，默认只渲染当前页，避免执行记录变多后切换页面和打开明细卡顿。
  - 支持按当前筛选时间范围删除明细；删除必须清理后端 `runs` 执行记录、对应工作区和产物目录，不得只是前端隐藏。
  - 删除明细不得回退或清零 AI产物清单累计调用次数。
- 美术执行台和 AI档案数据刷新规则：
  - `runs` 执行记录是动态队列数据，不等同于库存扫描缓存。
  - 进入美术执行台、`/agent-workers` 或 `AI档案` 时，必须轻量刷新 `/api/runs`，不能因为本地 `runs` 缓存已有数据就跳过后端刷新。
  - 轻量刷新执行记录不得触发库存扫描、Git 拉取、本地路径扫描或共享盘扫描。
  - 刷新后如果当前选中的执行记录已经被后端删除，必须自动切到最新可见执行记录或清空选择，不能继续显示本地旧记录。
  - 在 `/runs`、`/agent-workers`、`/ai-archive` 这类执行相关页面登录或刷新时，只恢复当前页面必需的轻量数据，例如 `runs`、`agentWorkers` 和账号列表；不得同时恢复库存、验证、成员快照、使用统计等重数据。
  - 美术执行台里的 `查看档案` 固定进入一级菜单 `AI档案`，并按当前执行记录 ID 精确筛选明细；不得跳回任务中心。
  - 美术执行台里的 `去人工验收` 才进入任务中心并打开关联业务任务验收。
  - 美术执行台和 AI档案之间切换时，不得触发库存扫描、Git 拉取、共享盘读取、整段日志渲染或任务链路大规模重算。
- 本地弹窗、提醒、LaunchAgent、轮询脚本：
  - 不得造成 Dock 反复闪烁、抢焦点、重复弹窗或窗口反复出现/消失。
  - 不得使用无限重弹的 `while true` 弹窗逻辑。
  - 同一轮多条待处理事项必须串行弹窗或写入待弹队列。
  - JXA/Cocoa 自定义窗口默认使用 `NSApplicationActivationPolicyAccessory`。
  - 弹窗必须固定尺寸和稳定布局，长内容使用滚动区域、列表或摘要窗口。

## 3.接口、数据结构、数据库约定

- 禅道同步范围：
  - 只看部门 ID `27`。
  - 精准抓取美术部门人员身上的任务。
  - 任务和 Bug 必须作为独立队列分开同步。
  - 任务同步失败不得影响 Bug 队列，Bug 同步失败不得影响任务队列。
  - 当前接入的 `zentao-mcp` 任务封装只确认支持按执行读取任务列表，例如 `GET /api.php/v1/executions/:id/tasks`；除非后续验证禅道接口支持，不得假设可以按部门或人员直接查询全部任务。
  - 当前禅道 REST 实测中，`/api.php/v1/tasks?assignedTo=...` 对部门 27 账号返回空结果，`/api.php/v1/bugs` 的 `assignedTo/openedBy/resolvedBy` 查询参数会被忽略；在未验证新接口可用前，不得把这些参数当作可靠的部门人员直查能力。
- 禅道字段保护：
  - 未经负责人明确要求，不得修改或清空父任务、关联需求、任务描述、标题、截止日、模块、任务类型、状态、预计工时、剩余工时等原字段。
  - 如确实需要改这些字段，必须由负责人明确要求并二次确认。
- 禅道同步按钮：
  - 任务大厅按钮文案为 `同步任务`，只触发任务同步和任务列表刷新。
  - Bug 大厅按钮文案为 `同步 Bug`，只触发 Bug 同步和 Bug 列表刷新。
  - 普通点击 `同步任务` 必须走快速任务同步，不得默认放大为全量历史扫描或同时同步 Bug。
  - 需要全量扫描时必须由负责人明确要求或另设独立入口。
  - 绿色完成提示必须以真实数据回填为准；同步刚触发时只能显示同步中或已开始同步。
- 禅道任务同步策略：
  - 日常 `同步任务` 必须走当前任务增量刷新，不得默认扫描执行列表再过滤美术人员名下任务。
  - 当前任务增量刷新只读取工作台本地已有、`source === 'zentao'`、存在 `taskNo` 且 `isCurrent !== false` 的任务号。
  - 当前任务增量刷新应按任务号逐个读取最新详情，例如 `GET /api.php/v1/tasks/:id`，并与本地缓存字段指纹对比。
  - 字段指纹至少覆盖任务标题、状态、指派人、截止日、描述、预计工时、剩余工时、消耗工时、进度、执行、需求、父任务、模块、优先级、更新时间等会影响负责人判断的字段。
  - 只有字段指纹变化时才更新本地任务缓存；没有变化的任务不得反复覆盖更新时间或制造无意义变更。
  - 如果任务已完成、关闭、取消或不再属于当前工作范围，必须标记为非当前或归档，不能继续混在当前任务列表。
  - 当前任务增量刷新只更新当前已有任务，不发现新任务，不同步 Bug，不扫描全量历史任务。
  - `同步 Bug` 必须走独立 Bug 队列；普通 `同步 Bug` 只刷新当前已跟踪 Bug 详情，不得默认扫描所有产品 Bug 后再过滤美术人员。
  - `发现新任务/全量扫描` 必须作为独立负责人操作，只在负责人明确触发时扫描执行列表、发现新执行或新指派给部门 `27` 成员的任务。
  - `发现新任务/全量扫描` 可以使用较慢的执行列表或产品 Bug 扫描逻辑，但必须有独立加载态、独立完成提示、独立失败提示和独立权限，不得伪装成普通 `同步任务` 或 `同步 Bug`。
- 数据展示与刷新：
  - AI产物清单、接入扫描、成员产物、资料库库存默认展示平台侧上次扫描缓存。
  - 进入页面、切换页面、切换账号、普通浏览器刷新、打开弹窗、修改 UI 样式或修改工作台配置时，不得重新抓取 Git、本地路径或共享盘。
  - 只有负责人点击 `刷新库存` 时，才重新扫描 Git、本地路径和共享盘。
  - Git 刷新流程固定：默认不拉 Git，只有点击 `刷新库存` 才拉取并扫描；失败保留旧缓存。
  - 本地路径和共享盘默认按上次缓存展示；断网、未连接服务器、路径暂时失效或 macOS 重启后未重新挂载时，不得让已扫描内容消失。
  - 修改工作台中的本地路径或共享盘路径，不得影响已经接入扫描的内容展示；新路径只影响后续负责人主动扫描或刷新。
  - 扫描失败、网络失败、权限失败、路径未挂载或接口异常时，必须保留当前已有清单和缓存，不得用空结果覆盖。
  - 如果服务端已有平台缓存，组员账号也应能读取同一份缓存数据。
- AI产物 Git/路径扫描：
  - 点击刷新库存必须抓取最新内容，不只是统计数量变化。
  - 刷新时必须识别新增、修改、删除；内容变化但数量不变也必须识别为变化。
  - 变化判断至少考虑 commit、提交时间、标题、描述、`preview` 正文、README/Skill 内容等有效指纹。
  - Git 扫描若发现产物内容有变更，质量分必须重新计算。
  - Git、本地路径、共享盘扫描都要把结果写入平台缓存，供负责人和组员共同查看。
  - 本地路径和共享盘扫描时，优先读取目标路径下的一级文件夹名作为产物名，并去重。
  - 如果文件夹内存在可识别 Skill 文件或二级 Skill 目录，需要摘取内容，并按 Git 扫描到的 Skill/md 展示方式补齐标题、说明、触发词和路径信息。
  - 来源记录默认使用上一次扫描结果，不实时刷新；只有负责人点击 `刷新库存` 或主动开始扫描，才更新来源记录。
  - 本地路径和共享盘断联时继续展示上次扫描缓存，不得清空产物列表。
  - 如果负责人删除了本地路径或共享盘里的文件夹内容，并在路径可访问时主动刷新，平台必须删除该文件夹产物关联的扫描信息、展示配置、贡献人覆盖、别名和隐藏配置，不得只做前端隐藏。
  - 删除后的路径内容如果后续重新新增，必须按新增接入重新扫描和建档。
  - 已作废或隐藏的产物不计入 AI 评分，但不得删除源文件或共享盘内容。
  - 修改贡献人只改工作台归属覆盖，不改源文件。
- AI产物调用次数：
  - 调用次数来源于成员同步上报数据和平台累计的产物使用统计，不来自页面假数据。
  - 成员上报的原始明细只作为短期明细缓存；平台必须另存累计调用次数。
  - 原始明细数据最多保留 2 天，超过 2 天要彻底清除，避免使用链接、打开页面或本机运行时卡顿。
  - 清除原始明细不得清零或减少原本调用次数；新数据来了，只在旧累计调用次数上增加次数。
  - 调用次数匹配必须同时看产物原名、展示名称、负责人修改后的别名、隐藏历史别名。
  - 路径扫描产物的产物名称和调用别名都必须参与调用次数匹配。
  - 工作台按钮操作名称如果命中产物名称或任一调用别名，且该操作产生了生成、执行、复用或调用结果，也必须增加该产物调用次数。
  - 调用别名匹配必须忽略大小写，并对空格、横线、下划线、中英文标点做统一标准化。
  - 调用次数归属必须绑定稳定产物键、产物原名、展示名称、当前别名和隐藏历史别名；负责人改别名后，旧累计调用次数不得丢失。
  - 同一次上报同时命中产物名、旧别名和新别名时，必须按事件唯一键去重，不得重复累计。
  - `usage-counters.json` 是累计调用次数事实来源；两天清理原始明细时只删除短期原始明细，不得删除或重置累计桶。
  - 删除 AI档案、清理原始上报明细、构建前端、刷新页面或切换账号，都不得导致已累计调用次数减少。
- AI产物调用别名数据约定：
  - 别名覆盖必须持久化到 `data/skill-version-overrides.json` 中的 `alias:<projectId>:<relativePath>` 记录。
  - 禁止只把别名写入 `owner:<projectId>:<relativePath>`、普通版本覆盖 key 或前端内存；读取历史数据时必须兼容并镜像到正确的 `alias:` 记录。
  - `aliases` 表示当前有效手动别名，用于界面展示和当前命中。
  - `aliasHistory` 表示隐藏历史别名，只用于历史调用次数归属和去重统计。
  - 保存别名后必须广播 `skill-version-overrides.changed` 平台事件；客户端只拉取 `/api/skill-version-overrides` 并在当前内存 `scans` 上打补丁，不得触发库存扫描。
  - 组员端读取别名必须和负责人端读取同一份服务端覆盖数据。
  - `skill-version-overrides.changed` 只刷新覆盖数据，不刷新库存、不读路径、不拉 Git。
- AI评分和有效占比：
  - `盛威` 的产物不计算有效占比。
  - 其他人的产物计算有效占比时，固定扣除管理员本人和 `盛威`，不是扣除当前登录账号对应人员。
  - AI评分必须基于当前平台缓存、产物归属、使用次数、验证/研究同步等真实数据计算，不得使用页面初始假数据。
  - `产物分计算说明` 必须保留。
  - 作废产物、隐藏产物必须从 AI 评分中排除；恢复后重新按正常规则计入。
  - 修复 AI 评分时，只改评分计算和展示相关逻辑，不得连带改动扫描缓存、贡献人覆盖、权限或刷新策略。
- 直接执行到 Figma 数据和执行边界：
  - 当前已定只实现第一场景：无需多轮对话，负责人或有权限账号选择一个 Skill/md，填写 Figma 链接，创建直接执行任务。
  - 直接执行任务的 `sourceType` 或 `executionMode` 为 `direct-skill`。
  - 直接执行任务必须记录 Skill/md 路径、Skill/md 内容快照、Figma 链接、写入方式、执行人、补充要求、创建人、领取设备、Worker 状态和执行日志。
  - 直接执行状态必须区分 `待领取`、`已领取`、`执行中`、`完成`、`失败`、`阻塞`。
  - 平台服务器只负责任务创建、队列记录、Worker 心跳、日志回传、状态展示和权限审计。
  - 平台服务器不得保存 Figma OAuth token，不得代替组员执行 Figma 写入，不得依赖工作台管理者 admin 的本机 Figma 环境。
  - 直接执行没有平台侧“开始”按钮；启动动作只能来自被指派执行人电脑上的本机 Worker 自动领取。
  - 组员本机 Worker 必须使用执行人自己的 Codex CLI、Figma MCP、Figma OAuth 和 Figma 文件权限。
  - 组员电脑默认没有负责人本机的项目代码，Worker 启动命令不得依赖负责人本机路径、相对 `scripts/` 路径或整套项目代码。
  - Worker 启动命令必须通过工作台服务 URL 下载 Worker 脚本到组员自己的用户目录；默认目录为 Windows `%USERPROFILE%\ArtDirectWorker`、macOS `$HOME/ArtDirectWorker`。
  - Windows `复制开机自启` 必须使用当前用户启动项安装 Worker，不得依赖管理员权限或必须注册系统计划任务；普通组员 PowerShell 可直接执行，重复安装只覆盖同一个启动项。
  - Windows `复制开机自启` 重复安装时必须停止同一 Worker 目录下的旧进程，并清理当前用户启动项里旧的 `ArtDirectWorker-*` 快捷方式，只保留本次账号对应的启动项；不得因为检测到旧进程就继续沿用旧账号、旧密码或旧工作台地址。
  - macOS `复制开机自启` 使用当前用户 LaunchAgent；开机自启安装只需要操作一次，后续电脑登录系统自动启动 Worker，除非脚本、账号或工作台地址需要更新。
  - 直接执行 Worker 必须优先使用平台随任务下发的 Skill/md 内容快照，不得要求组员电脑能读取负责人本机 Skill/md 文件路径。
  - 直接执行 Worker 默认运行在执行人自己的 `ArtDirectWorker` 目录，该目录不保证是 Git 仓库或 Codex 已信任目录；Worker 拉起 `codex exec` 时必须带 `--skip-git-repo-check`，避免还未真正执行 Figma 写入就因目录信任检查秒失败。
  - Figma 必须走原生授权和 Figma MCP；组员登录工作台后，也必须在自己的电脑使用自己的 Figma 账号和本机 Figma MCP。
  - 如果执行人本机 Codex 不可用、Figma MCP 未配置、Figma OAuth 失效、目标 Figma 文件无编辑权限或工具列表缺少写入工具，Worker 必须停止执行并回传阻塞原因。
  - 只有 Figma 写入工具真实返回 `createdNodeIds` 或 `mutatedNodeIds`，才允许判定写入完成。
- Web 端 Codex 对话数据约定：
  - Web 对话入口复用普通执行权限 `run.codex.execute`，只对非 `direct-skill` 执行记录开放；直接执行仍由执行人本机 Worker 领取，不走平台对话启动。
  - Web 对话新增执行必须调用 `/api/runs/:id/retry` 创建新执行，再通过 `/api/runs/:id/start` 启动；不得直接在当前记录上覆盖结果。
  - `/api/runs/:id/retry` 请求体必须使用字段白名单，只允许 `title`、`requirement`、`figmaLinks`、`showdocHints`、`targetPage`、`stage`、`codexRequest` 这类追加执行字段覆盖到新执行；不得允许前端传入任意字段覆盖项目、权限、状态、执行人或路径。
  - `codexRequest` 只允许保存 `model`、`reasoningEffort`、`requestStandard`、`source`；`requestStandard` 必须限制长度，不能保存完整聊天历史、大日志、API Key、Figma token 或本机敏感路径。
  - 后端启动普通执行时必须将全局 Codex 配置和 `run.codexRequest` 合并：API Key、Base URL 和 provider 来自全局配置，单次 `model` 和 `reasoningEffort` 可覆盖本次启动参数。
  - `reasoningEffort` 允许值必须受控，例如 `low`、`medium`、`high`、`xhigh`；不得把未校验字符串直接拼入 Codex CLI 参数。
  - Web 对话默认不新增轮询、不新增持续心跳、不保存多轮全量消息流；只在用户展开并提交时创建一条新执行，避免卡顿和数据暴增。
- Worker 数据与接口：
  - Worker 心跳写入 `agent-workers` 数据，至少包含执行人、设备、最近心跳、Codex 是否就绪、Figma MCP 是否就绪、能力标记和检查消息。
  - Worker 设备花名保存为独立展示字段；后续心跳不得因未携带花名而清空已保存花名。
  - Worker 设备花名只能由设备所属账号本人修改，不允许组员修改他人设备花名。
  - 直接执行领取必须校验执行人账号、项目权限、被分配关系、本机能力和 Worker 权限。
  - Worker 只能领取并回传分配给自己的直接执行任务；除 admin 审计能力外，不得回传他人任务状态。
  - Worker 日志回传要限制单次长度，避免日志过大拖慢平台。
  - 工作台前端不得整段渲染原始 Codex/Worker/Figma 日志；`run.log` 中的 Figma 截图、base64、图片、超长 JSON 和大段 MCP 工具返回必须摘要、截断或过滤后再展示。
  - `/api/runs/:id/log` 默认只返回尾部日志摘要；完整原始日志只作为服务器文件保留，排查时查看文件，不得作为页面切换或刷新时的默认渲染数据。
  - 前端默认请求执行日志尾部不得超过 `48KB`；如需排查更大原始日志，只能在明确调试入口或服务器文件中查看，不能作为页面默认加载内容。
  - 美术执行台的历史原始执行日志默认收起；选择已完成/失败/取消的执行记录时不得自动读取历史日志，只有任务运行中或用户点击日志入口时才读取尾部摘要并渲染。
  - 美术执行台的原始执行日志入口固定放在右侧详情顶部操作区 `删除` 按钮右侧，表现为和全局 `操作日志` 一致的图标按钮；不得再放在详情底部折叠块里。
  - 美术执行台点击原始执行日志图标后，必须从右侧半屏 `ElDrawer` 滑出查看内容；抽屉必须直角，内容区域独立滚动，只渲染安全摘要和尾部日志。
  - 美术执行台的原始执行日志收起状态不得提前执行 Markdown 渲染，只有用户点击日志入口打开抽屉时才渲染安全摘要，避免切换页面和刷新卡住。
  - Worker 心跳不得高频写入数据：默认兜底任务轮询间隔为 5 分钟，默认最近心跳刷新间隔为 5 分钟，前端在线阈值为 10 分钟；空轮询无任务时不得反复写入 `agent-workers` 状态文件。
  - 平台创建或更新直接执行时，必须通过实时事件唤醒在线 Worker 立即检查任务；5 分钟轮询只作为实时事件断线或网络抖动后的兜底。
  - Worker 启动后必须先上报一次准备状态；后续只在定时心跳、实际领取任务或执行结束后刷新 Worker 状态。
  - Worker 不能依赖组员浏览器网页是否置顶或是否正在操作工作台；只要组员电脑启动 Worker 且登录成功，就必须能接收直接执行任务。
  - 工作台前端的 Worker 状态自动刷新必须克制：只有浏览器标签页可见，且当前在美术执行台或本机执行状态页时，才因 `agent-workers.changed` 事件刷新 Worker 状态。
- 权限：

  - 负责人/admin 默认拥有管理类权限；组员默认只拥有必要读取、明细查看和被授权的执行能力。

  - 与直接执行相关的权限必须拆分，不得全部复用 `api.runs.execute`：
    - `menu.agentWorkers`：查看本机执行状态页。
    - `menu.aiArchive`：查看 AI档案页。
    - `run.directSkill.create`：创建直接执行。
    - `run.directSkill.workerCommand`：复制 Worker 手动启动或开机自启命令。
    - `api.agentRuns.create`：创建直接执行 API。
    - `api.agentWorkers.read`：读取 Worker 状态 API。
    - `api.agentWorkers.heartbeat`：Worker 心跳 API。
    - `api.agentWorkers.alias`：本人修改 Worker 设备花名 API。
    - `api.agentRuns.claim`：Worker 领取直接执行 API。
    - `api.agentRuns.log`：Worker 回传日志 API。
    - `api.agentRuns.status`：Worker 回传状态 API。
    - `api.aiArchive.delete`：AI档案范围删除 API。
  - `menu.runs` 只代表美术执行台页面；`menu.agentWorkers` 只代表本机执行状态页面；`menu.aiArchive` 只代表 AI档案完整明细页面，三者不得混用。
  - `api.runs.execute` 保留给普通执行任务创建、启动、中断等旧执行能力；直接执行创建和 Worker 流程必须使用专门权限。

## 4.禁用写法、已定功能、固定执行命令

- 禁止事项：
  - 禁止未经要求重构业务结构。
  - 禁止顺手修改禅道字段。
  - 禁止扫描失败、路径断联、网络失败或接口异常后用空数据覆盖缓存。
  - 禁止页面进入、普通刷新、打开弹窗或修改工作台配置时自动重新抓取库存。
  - 禁止刷新页面后把 AI产物清单置空或要求重新接入扫描。
  - 禁止把 AI产物刷新做成只看数量变化。
  - 禁止组员端因为无扫描权限而看不到平台缓存。
  - 禁止作废产物或隐藏产物继续计入 AI评分。
  - 禁止删除源文件或共享盘内容来实现作废、隐藏或工作台展示调整。
  - 禁止共享盘或本地路径断联时清空已扫描产物。
  - 禁止把接入扫描弹窗做成实时刷新库存入口。
  - 禁止本地路径或共享盘内容删除后只做前端隐藏而不清理扫描关联信息。
  - 禁止成员上报原始明细清理时清零累计调用次数。
  - 禁止因为负责人修改产物别名而丢失旧累计调用次数。
  - 禁止把一次调用因命中多个别名或名称重复计数。

  - 禁止只在当前浏览器内存里更新别名而不写服务端覆盖数据。
  - 禁止通过刷新库存、重扫 Git、读取共享盘或读取本地路径来实现别名实时同步。
